# Adversarial design review (workflow) — verdict: sound_with_fixes

## oracle_upper_bound
Yes, oracle-hint (inject the TRUE first-op type) is the correct primary kill/support gate, and the right comparison is oracle-hint vs no-hint at matched sampling (same k, temperature, seeds, no-think, same behavioral-depth slice). Kill result: Δcoverage@k ≈ 0 and Δgreedy@1 ≈ 0 -> the first op is not the bottleneck, so NO readout (probe or otherwise) can help; the approach is dead and the probe never needs to run. Support result: oracle-hint >> no-hint, establishing headroom the probe can chase. BUT there is a real reason a correct first-op-TYPE hint can fail even when the first op IS the bottleneck: the C19 probe decodes op TYPE only, not the PARAMETER (e.g. "shift" but not by how much). If the model needs the argument bound, a type-only oracle will look inert and falsely kill the whole idea. FIX: run TWO upper bounds -- oracle-type (matches what the probe can deliver) and oracle-full-op (type+args). Interpretation: oracle-full helps but oracle-type does not => bottleneck is parameter-binding, which the probe structurally cannot address -> approach dead for the RIGHT reason. Both null => first op genuinely not the lever. Also add a manipulation check on the oracle arm: measure P(generated first op == hinted op) vs no-hint; if the model does not increase first-op compliance under a true hint, it is ignoring the hint and the paradigm fails trivially before any probe question is meaningful.

## giving_away_the_answer
At depth-1 the first op IS the whole answer, so probe-hint trivially solves it -- correctly relegated to a sanity check only. At depth-2/3 the first op is 1/2 or 1/3 of the composition, so a partial hint. Whether a probe-hint gain is genuine LATENT self-elicitation vs answer-leak hinges on ONE thing the current design does not pin down: is the probe reading a first-op that the MODEL computed internally, or is it an external linear solver reading the first-op straight off the I/O examples encoded in the prompt activation? Per C19, the probe is fit on last-prompt-token activations, and those activations encode the visible I/O pairs, which fully determine the first op. A linear classifier that recovers the first op from surface I/O structure and feeds it back is closer to solving a sub-problem FOR the model than to eliciting the model's own latent computation. The control that isolates this is C19's layer-0/embedding probe: C19 reports embedding-probe at chance (0.07/0.07/0.04) while mid-layer rises to signal, which is exactly the evidence that the first op is COMPUTED by the network (post-embedding) rather than surface-readable. That control MUST be re-run on the fresh fsig-disjoint eval set and reported: if a layer-0/raw-embedding probe already decodes the first op above chance on the eval tasks, then probe-hint is external information injection, not self-elicitation. If layer-0 stays at chance and only the mid-layer (post-computation) probe decodes it, the info was in the model's own representation and externalizing it is legitimate latent elicitation. Second isolating control: the probe injects NO oracle label at inference -- its weights are a generic activation->op map learned from disjoint labeled tasks, and the per-eval-task prediction comes entirely from that task's activation. To make the self-elicitation quantitative, pre-register that probe-hint gain should approximate probe_accuracy x (oracle-type-hint gain), and decompose probe-hint outcomes by probe-correct vs probe-incorrect subsets: genuine elicitation predicts gain concentrated on the probe-correct subset and roughly zero (or negative) on probe-incorrect. If probe-hint helps EVEN on probe-incorrect tasks, the gain is a prompt artifact, not readout content.

## probe_generalization
This is the highest-risk area and the design currently imports C19's numbers (d2 0.42, d3 0.13 real) as if they transfer -- they very likely do not, for two concrete reasons in probe.py. (1) C19's held-out accuracy is a within-pool stratified-by-CLASS split (probe.py L19-40): train and test tasks come from the SAME generated pool, so shared task-family structure can leak across the split. The new eval demands a stricter fsig-DISJOINT split, on which accuracy will generally be lower. (2) C19 selects best_L = argmax over layers of the TEST accuracy itself (probe.py L68), i.e. best-layer chosen on the evaluation set -- an optimistic selection over ~L+1 layers, so the reported 0.42/0.13 are upward-biased even within-pool. FIXES: (a) fit standardizer+PCA+logistic AND choose the best layer on a train/val split only, freeze layer per depth, then report accuracy on the fsig-disjoint eval tasks as a FIRST-CLASS gating number, not an inherited constant. (b) Confirm fsig disjointness programmatically (no eval fsig appears in train) and confirm the label space matches -- capture.py:22 uses only the list-family's ~16 prims, so eval must be the same family or the 16-way probe is undefined. (c) Report probe eval accuracy ABOVE majority-class baseline (d3 majority chance is ~0.14, high due to imbalance) so a probe that mostly predicts the majority op is not mistaken for signal. (d) Capture the eval activation under the SAME prompt the probe was trained on (C19's identification prompt); if the four generation arms use a different prompt than the probe's training prompt, distribution shift will degrade probe accuracy below the reported figure. The headline hint-effect analysis must be conditioned on this measured eval accuracy, and if fsig-disjoint eval accuracy collapses toward chance at d2/d3, the probe-hint arm is expected to be inert and that must be stated up front as a predicted outcome, not a surprise.

## hint_format_and_controls
Hint-as-prompt-line is fine but under-controlled as specified. wrong-hint: make it RANDOM-incorrect op (uniform over wrong ops) -- that is the correct content-causality control (correct helps, wrong hurts => CONTENT matters, mirroring C28's T_corrupt logic). Do NOT make wrong-hint the probe's specific mistakes; the probe's mistakes are already folded into the probe-hint arm and are better characterized by the probe-correct/incorrect decomposition (see giving_away_the_answer). MISSING CONTROL (must add): a NEUTRAL/placebo hint -- a syntactically matched but uninformative line (e.g. "the first operation is one of the listed primitives"). no-hint vs neutral-hint isolates the pure prompt-FORMAT effect (does adding any instruction line change formatting/coverage); it must be ~0 or every arm is confounded. Prompt-sensitivity: instrument compliance P(generated first op == hinted op) per arm -- if oracle-hint does not raise compliance, the model ignores hints; if wrong-hint drives compliance up and solve-rate down, the model over-trusts, which is itself the causal signal. Metrics: report BOTH coverage@k (curve vs k, not a single k) and greedy@1, per BEHAVIORAL depth, with matched sampling (identical temperature and seeds across arms) so the hint's entropy reduction does not silently move coverage@k independent of content. Pre-register the effect bar and SE: C20 died because a +0.05 effect at n=150 fell under a +0.10 bar, so state n-per-depth and the margin before running. Add the direct externalization metric: fraction of generations whose first op is correct, by arm -- this measures whether the readout got EXPRESSED even when the full task is not solved (critical at d3, where a correct first op still leaves an unsolvable depth-2 residual). The pattern that = "externalization elicits where steering (C20) failed and selection (C17) added nothing": ordering oracle >= probe > no-hint on greedy@1 AND coverage@k; probe-hint gain concentrated on the probe-correct subset and ~ probe_acc x oracle-gain; wrong-hint <= no-hint; neutral-hint ~ no-hint; layer-0-decoded-hint ~ chance-hint; graded by depth (elicits at d2 where headroom 0.42, fades at d3 where signal is a 0.13 thread).

## Confounds
- Nominal vs behavioral depth: sibling decompose_compose_frontier found 40% of nominal depth-3 tasks collapse to <=2; must slice all results by min_depth_leq (families.py:130), not nominal depth, and confirm the eval generator applies that filter (make_task L188 does; the frontier generator did not).
- C19 best-layer selected by argmax on the test set (probe.py:68) inflates the expected probe accuracy the design inherits.
- C19 held-out split is within-pool stratified-by-class (probe.py:19-40), not fsig-disjoint; fresh-task eval accuracy will likely be lower and must be re-measured.
- Surface-I/O decodability: a linear probe may read first-op off the I/O examples in the activation rather than the model's computation -> external solver, not latent elicitation; layer-0/embedding control on eval set disambiguates.
- Prompt-format effect of adding any hint line (needs neutral/placebo arm).
- Hint reduces generation entropy, which can move coverage@k independent of content (needs matched sampling + coverage-vs-k curve).
- Type-only vs parameter hint: probe decodes op TYPE only; if the bottleneck is parameter-binding, type hints (probe and oracle-type) are inert even though first op matters (needs oracle-full-op upper bound).
- Activation distribution shift: probe trained on identification-prompt activations but arms may generate under a different prompt.
- Class imbalance: d3 majority-class chance ~0.14; report probe acc above majority and guard against majority-op hint gains.
- Label-space/family mismatch: probe label space is list-family's ~16 prims (capture.py:22); eval must use the same family.
- Model ignoring or over-trusting the hint (needs compliance manipulation check).

## How addressed
