# Adversarial design review (workflow) — verdict: sound_with_fixes

## param_surface_readability
NO — the layer-0 concrete probe as written cannot distinguish "param is latent in the model" from "param is surface-readable from I/O," because it is DEGENERATE, not a valid control. Activations are captured at the LAST real prompt token (gen_lib.py:222), and the prompt ends with a fixed template string ("...Only a ```python code block."). Qwen is RoPE, so hidden_states[0] (embedding output) carries NO additive positional signal and NO cross-token I/O — it is the embedding of one fixed token, IDENTICAL across every task. A logistic probe on a constant input predicts the majority class, so conc0_acc will read ~base-rate/chance for a reason that has nothing to do with whether the param is surface-readable. It cannot detect surface-readability, so "layer-0 near chance" here is NOT evidence the param is model-computed. (C19/C30's "chance at embedding" argument for the type inherits the same weakness; the meaningful signal there was the RISE via attention, not the layer-0 floor.)

A genuine surface control must READ the I/O: either (a) an EXTERNAL classifier (logistic/kNN) fit on raw numeric I/O features per task — input/output lengths, sums, min/max, element-wise diffs of the visible examples — with NO 4B forward pass, or (b) a layer-0 probe POOLED over the I/O-example token span (not the last template token). If that external-on-I/O baseline already reaches the mid-layer concrete accuracy (and the mid-layer param|type), the param is surface-trivial and the "latent readout" framing is unsupported; if the mid-layer probe beats it substantially, the model has genuinely distilled the param.

Does surface-vs-computed matter for the DEPLOYABILITY claim? For raw deployability, NO — a hint that lifts greedy@1 helps regardless of provenance. But for the MISSION claim (elicit latent capability already in the fixed weights, beat sample-more via training-free readout of the MODEL), it matters decisively: if a trivial external linear solver on I/O produces the same hint, you have externalized feature-engineering, not model-latent elicitation, and the honest framing changes. Report BOTH, separately labeled: (1) deployability = does probe_full lift greedy@1 on param-first-op tasks; (2) latent-vs-surface = mid-layer concrete/param|type vs the external-I/O baseline's concrete/param|type. If the external solver also deploys, disclose that the cheaper non-model pipeline suffices.

## param_decodability_metric
P(concrete correct | type correct) vs mean 1/nparams is a REASONABLE conditional but not sufficient alone as "param decodability," and has three issues. (1) Selection/collider bias: conditioning on type-correct (defined via the same 32-way probe's argmax type, run_hints_full.py) selects easy, clean-representation tasks, which are plausibly also the tasks where the param is easiest — so the conditional OVERSTATES intrinsic param decodability relative to a random task. It answers "given the type is nailed, is the param also there," which is a fair question, but it is not an unbiased estimate; pair it with the UNCONDITIONAL P(concrete correct on param tasks) and the raw type acc so the number is interpretable. (2) Power: first ops are uniform over 16 TYPES, so only ~37.5% of eval tasks are param-bearing (~45/depth), and type-correct trims that to maybe ~15-25 tasks, then pooled across 6 param types with different nparams (6/3/3/4/3/3). CI on a proportion from n~18 is roughly +/-0.23 — too thin for a headline "is the param decodable" claim. Report per-type counts and a binomial test against the per-task 1/nparams chance; treat pooled param|type as exploratory. The activation forward passes are CHEAP (no generation), so estimate all probe/param-decodability metrics on a MUCH larger eval set (e.g. 800-1000/depth) decoupled from the expensive generation arms (120/depth). (3) The chance denominator: 1/nparams assumes uniform param prior (make_task does pick params uniformly, so OK) — fine, but a stronger complementary baseline is a param-only MAJORITY predictor per type. Missing complementary metric that is actually the crux: param|type is NOT computed for the surface/layer-0 baseline — so the design cannot compare mid-layer param|type against a surface param|type, which is exactly the "latent vs surface" test for the PARAM specifically. Add param|type for the (fixed) external-I/O surface baseline. Also consider normalized accuracy-above-chance or MI for cross-type aggregation instead of pooling heterogeneous nparams.

## probe_full_delivery
The bound "probe_full gain ~= concrete_acc x oracle_full_gain, concentrated on concrete-correct tasks" is the right FIRST term but the pre-registration is INCOMPLETE: it omits the wrong-hint term. On concrete-INCORRECT tasks (fraction 1-concrete_acc) probe_full delivers a confidently-stated WRONG concrete op = the wrong_full condition, which for a full op+param hint can actively mislead greedy decoding (below no-hint). The correct pre-registered decomposition is: probe_gain ~= concrete_acc x oracle_full_gain + (1-concrete_acc) x wrong_full_gain, where wrong_full_gain is measured (likely <= 0). This makes the net-positive threshold: concrete_acc x oracle_gain > (1-concrete_acc) x |wrong_penalty|; with oracle_gain ~0.16 (0.03->0.19) even modest concrete_acc nets positive IF wrong hints are near-neutral, but if wrong hints strongly mislead the required concrete_acc rises — this must be COMPUTED from the arms, not assumed. The design already logs conc_correct and per-arm greedy per task, so the two-term model is directly verifiable: check probe_full ~= oracle_full on conc-correct tasks and ~= wrong_full on conc-incorrect tasks. Ceiling/control choices: replicating oracle_full IN-HARNESS as the ceiling is correct (do NOT rely on C30's 0.19 across harnesses; pre-register a sanity gate that oracle_full greedy@1 depth-2 ~ 0.19, else harness mismatch). wrong_full (random wrong concrete) is a fine generic content control but it changes BOTH type and param, so it does NOT isolate the PARAM — the experiment's whole thesis. Add two arms: oracle_type (true TYPE only, no param) so oracle_full - oracle_type = the param's marginal deployable value in-harness; and wrong_param (true type, WRONG param) to show getting the param wrong specifically hurts. Without these you cannot attribute probe_full's lift to the PARAM rather than to re-confirming the type. Finally, because 62.5% of first ops are non-param (where full_op_repr collapses to a bare type hint), the HEADLINE deployable lift must be reported SEPARATELY for param-first-op vs non-param-first-op tasks (is_param is logged) — otherwise the "param deploys" claim is diluted by non-param tasks where only the type is at play. The money metric is oracle_full/probe_full lift ON param-first-op tasks, ideally net of oracle_type.

## Confounds
- Degenerate surface control: layer-0 concrete probe reads a constant last-template-token embedding (RoPE, fixed prompt tail) -> ~chance for reasons unrelated to surface-readability; cannot support any latent-vs-surface conclusion.
- Non-param first ops (62.5% of tasks) collapse the 'full op' hint to a bare type hint, so an aggregate greedy@1 lift confounds the param story with pure type hints; must split by is_param.
- wrong_full changes both op-type and param, so it cannot isolate the param; a right-type/wrong-param arm is missing.
- First-op non-identifiability: make_task only guarantees no shallower pipeline reproduces the I/O, not that the first (op,param) is the UNIQUE valid first op; a different concrete first op may be extendable to the same I/O. This can under-count concrete_acc, and wrong_full may occasionally BE a valid alternative first op, weakening it as a pure content control.
- param|type conditions on the same probe's type-correctness (collider/selection bias toward easy tasks) -> overstates intrinsic param decodability; unconditional P(param) not reported alongside.
- Thin n: ~37.5% param-first-op x type-correct fraction leaves ~15-25 tasks for the headline param|type estimate; CIs are very wide; generation-set n (120/depth) is coupled to cheap probe-eval n.
- Concrete vocab and nparams are derived only from training first-ops; if any of the 32 concrete ops is absent from the 1200 train tasks it is unpredictable AND its 1/nparams chance denominator is wrong (not asserted).
- oracle_full ceiling assumed from C30 (0.19) rather than gated in-harness; harness/prompt-format drift could silently move the ceiling.

## Must-fix
- Replace the degenerate layer-0 last-token surface control with a genuine surface baseline: an external classifier on raw numeric I/O features (lengths/sums/min-max/element-wise diffs), plus optionally a layer-0 probe pooled over the I/O-example span. Compare mid-layer concrete acc AND param|type against it.
- Compute param|type (P(concrete|type correct)) for the surface baseline too, not just the mid-layer probe — this is the actual latent-vs-surface test for the PARAM.
- Report the deployable greedy@1/coverage@6 lift SEPARATELY for param-first-op tasks vs non-param-first-op tasks (is_param already logged); make param-first-op tasks the headline.
- Add an oracle_type arm (true type, no param) so param marginal deployable value = oracle_full - oracle_type is measured in-harness; add a wrong_param arm (true type, wrong param) to isolate the param as content control.
- Fix the pre-registered decomposition to the two-term model: probe_gain ~= concrete_acc*oracle_full_gain + (1-concrete_acc)*wrong_full_gain; verify it per-task from the logged conc_correct + per-arm greedy, and compute the net-positive concrete_acc threshold instead of assuming it.
- Gate on in-harness oracle_full replication (pre-register oracle_full depth-2 greedy@1 ~ 0.19; abort/flag if the ceiling doesn't reproduce).
- Decouple probe-metric n from generation n: estimate concrete_acc, type_acc, and param|type on a large cheap forward-pass set (>=800/depth) with per-type counts and binomial CIs vs 1/nparams; keep generation arms at 120/depth.
- Assert the concrete vocab contains all 32 concrete ops and that per-type nparams matches the DSL, else handle/flag missing classes (affects both predictability and the chance denominator).
- Report the unconditional P(concrete correct on param tasks) alongside param|type, and acknowledge the type-correct conditioning bias.
- Acknowledge/mitigate first-op non-identifiability: at minimum verify wrong_full hints are not valid alternative first ops for the task I/O (or note the caveat) so wrong_full and concrete_acc are interpretable.
