# Adversarial design review (workflow) — verdict: flawed

## fairness_vs_sample_more
Not fair as worded. The decomposition arm gets four things "sample more" lacks: (a) the DSL menu spelled out with English descriptions, (b) the interpreter materializing each intermediate state so the model only ever answers a 1-of-32 "current->target" question instead of writing a 3-op program, (c) the interpreter as an unlimited verifier against the examples, and (d) the beam+distance search structure that does the actual composition. So P2 is a tool-augmentation claim, not a latent-capability claim — beating single-shot is expected because depth-3 has been externally decomposed into three depth-1 decisions with the interpreter carrying state. This is precisely what the sibling frontier experiment concluded ("the crack comes from the composition structure + interpreter, not the model's planning"). To make it honest: (1) give the single-shot baseline the identical DSL menu in its prompt; (2) give single-shot the interpreter too — the fair "sample more" is M2-style sample-k-programs + execute-against-visible + select (the frontier already ran this and got 0.125); (3) budget-match on BOTH model forward-passes AND interpreter/verifier calls — set single-shot's k equal to the search's total interpreter-call budget, because each interpreter verification is information the search consumes and sample-more does not; report the full coverage-vs-interpreter-call curve, not a point. Under a genuinely budget-matched, DSL-matched, interpreter-equipped comparison the frontier already showed the model does not win on coverage.

## pruning_does_the_work
The random-guided ablation does NOT isolate the model's contribution, and worse, it is the wrong control — it is a floor, not the honesty bar. The honest bar (already established in the sibling experiment) is budget-matched exhaustive/brute-force proposals (all 32 ops), which MATCHED the model on depth-3 coverage (0.43 vs 0.40); the model only made search cheaper. Beating random is trivially true and proves nothing about latent capability. Separately, the distance-to-target pruning is itself an interpreter-side oracle heuristic that can carry the composition independent of the model, so the real confound is uncontrolled. Required ablation is a 2x2: proposals in {model-ranked, uniform-random, brute-all-32} x beam in {distance-pruned, unpruned}. If random-proposals + distance-pruning already cracks depth-3, the pruning does the work and the model is irrelevant. Also define "distance-to-target" precisely (element-wise on which examples?) — many DSL ops are non-monotone in any elementwise distance, so if it works it is doing real search. Bottom line: replace P3's random control with budget-matched brute-force as the primary control (keep random only as a floor), and add the pruning-off ablation.

## proposal_mechanism
Likelihood-ranking a closed 32-op set is a much easier task than generation and leaks the answer space: the model is handed the menu and only has to rank, so "proposal skill" collapses to a 1-of-32 multiple-choice question. With top-k up to 8 of 32 you retain ~25% of the branching — barely a prune — and the frontier already found the right primitive is in the top-8 only ~2/3 of the time at the root and degrades over steps, so the closed-set ranking is a weak signal that the beam+pruning+interpreter compensate for. The honest test of latent step-proposal skill must (1) report top-1 ranking accuracy against the ground-truth next op and the top-k needed to hit target coverage (if k must be large, the "skill" is weak); (2) compare model-ranking head-to-head with a pure interpreter-distance ranking at matched top-k — if distance-ranking matches model-ranking, the model adds nothing; (3) include a free-generation variant (model emits the op name with no letter-logit menu) to see whether the proposal skill survives without the closed-set scaffold, since letter-logit-over-a-given-menu is not the deployable/generative form of the capability the claim is about.

## latent_capability_claim
Success on P1+P2+P3 as designed does NOT legitimately show depth-3 composition is latent in the fixed weights; the weaker, better-supported reading is that tool-augmented SEARCH (composition structure + interpreter + distance pruning) externalizes the depth-3 computation while the model only ever solves depth-1 sub-problems. Latent depth-3 composition would require the model to compose three steps internally; here the search composes them and the interpreter holds state, so the depth-3 wall "relocates to the planner" (the sibling experiment's exact finding). The extra controls that would be needed to even attempt the latent claim — budget-matched brute-force (not random), min-depth-verified TRUE depth-3 tasks, high per-step top-1 ranking accuracy — are the ones the frontier already ran, and they came out AGAINST the latent reading (model ties brute on coverage; only 4/24 true-depth-3 solved). The one piece of genuine "into-the-weights" evidence in this arc is banking the search-found solutions via QLoRA lifting monolithic depth-3 — but that involves training, which this experiment forbids. So training-free search shows an orchestration/elicitation win for SEARCH, not that depth-3 is latent in the frozen forward pass. To nail (or refute) latent-ness you must beat budget-matched brute-force on coverage on min-depth-verified true-depth-3 tasks; absent that, claim only "reachable by tool-augmented decomposition," not "latent."

## Confounds
- Task collapse: ~30-40% of nominal depth-3 tasks (including this design's C23/C24 n=80 set, ~30% collapsed) are behaviorally depth <=2 via shallower-equivalent compositions (e.g. sort_asc.reverse == sort_desc); 'depth-3 solved' is inflated unless BFS min-depth-verified
- Distance-to-target pruning is an interpreter-side oracle heuristic that can perform the composition independent of the model — not isolated by a random-proposal ablation that keeps the pruning on
- Information asymmetry: decomposition gets the DSL menu, per-step intermediate-state feedback, and an unlimited interpreter verifier; single-shot 'sample more' gets only I/O and must write whole programs
- Budget accounting: interpreter/verifier calls are unpriced information; measuring only model forward-passes hides that the search consumes hundreds of interpreter oracle-checks per task
- Closed-set 1-of-32 likelihood-ranking is far easier than generation and with top-k=8 retains ~25% of branching, so the beam/pruning/interpreter (not the model) can carry the search
- Wrong primary control: random-guided is a floor, not the honesty bar; budget-matched brute-force (already shown to match the model on coverage) is the correct comparison
- Held-out leakage into the search: terminating on 'matches all visible+hidden examples' uses the hidden grading set inside the search; termination/verification must use VISIBLE examples only, then grade generalization on hidden

## Must-fix
- Replace P3's random-guided control with budget-matched BRUTE-FORCE (all-32 proposals) as the primary honesty bar; keep random only as a floor. The claim must beat brute-force on COVERAGE (not just efficiency) to support latent-ness — the sibling frontier experiment already found it does not.
- Add a distance-pruning ablation: 2x2 of {model, random, brute} proposals x {distance-pruned, unpruned} beam, to prove the model — not the interpreter distance heuristic — is doing the work.
- Restrict all depth-3 (and depth-4) reporting to BFS min-depth-VERIFIED true-depth-3 tasks; the C23/C24 n=80 set is ~30% collapsed. Report true-depth-3 coverage separately from collapsed.
- Fix the search termination/verification to use VISIBLE examples ONLY; grade generalization on hidden. As written ('matches all visible+hidden = solved') the search peeks at the held-out set.
- Make 'sample more' fair: give the single-shot baseline the same DSL menu AND the interpreter (sample-k + execute-on-visible + select), and budget-match on interpreter calls, not just forward-passes.
- Report per-step top-1 ranking accuracy vs ground-truth next op and the top-k required; add a free-generation proposal variant. If top-1 is low or a large top-k is needed, the 'latent step-proposal skill' is weak and the interpreter is carrying it.
- Reframe the headline: absent a win over budget-matched brute-force on true-depth-3, claim 'depth-3 is reachable by training-free tool-augmented decomposition (search + interpreter)', NOT 'depth-3 composition is latent in the frozen weights' — the wall relocates to the planner, it does not vanish.

## How this experiment RESPONDED (pivot)
The review (correct) showed the original 'is depth-3 latent' framing is flawed and that the sibling C12 already ran the base-model decomposition search (guidance = efficiency not coverage; brute matches it). PIVOT: (a) fixed the peeking bug -- search now terminates on VISIBLE examples only, grades hidden separately; (b) brute-force (all 32 ops) is the PRIMARY honesty bar, random only a floor; (c) all 80 held-out tasks are min-depth-VERIFIED true-depth-3 (families.py collapse-rejects; the ~30% collapse was C12's DSL, not this one); (d) added the 2x2 pruned/unpruned ablation; (e) NEW contribution beyond C12: per-step ground-truth next-op RANKING ACCURACY (localizes the planner-wall as a LOOKAHEAD wall -- step-3 recognition intact, step-1/2 at chance) and whether BANKING (C24 adapters) lifts lookahead or is monolithic-only. Claim is scoped to 'the model's step-proposal is depth-1 recognition without lookahead; banking installs compilation not planning' -- NOT 'depth-3 is latent'.
