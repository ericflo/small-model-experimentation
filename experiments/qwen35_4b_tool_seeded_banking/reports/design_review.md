# Adversarial design review (multi-agent workflow)

**Verdict:** sound_with_fixes

## Confounds
- SUBSTRATE-VOCABULARY MISMATCH (biggest confound): the C12 Decomposer/run_search is hardwired to gen_tasks.PRIMS (23 primitives: adds filter_even/odd, keep_positive, running_max, filter_gt_k, filter_lt_k, chunk_sum_k) with wider PARAM_OPTS (mod_k [2,3,4,5] vs families [2,3,4]; add_k+5; mul_k+-1), whereas eval_ladder/C21 use families.LIST_PRIMS (16 prims). A richer search toolset can crack a nominally-depth-3 wall_climbing task with a SHORTER or alternative decomposition, and trains the banked model to emit out-of-family ops. This conflates '4B reaches depth-3' with 'a bigger DSL reaches it,' and a 'true-depth-3' task in the 16-op DSL may be depth<=2 in the 23-op DSL, spuriously inflating both the harvest yield and banked depth-3 coverage. Search vocabulary+params MUST be locked exactly to families.LIST_PRIMS.
- BRUTE-FORCE CONTAMINATION OF THE SEED SET: run_search.py saves found_solutions as the UNION of guided AND brute-force hidden-generalizing solutions. Banking brute seeds credits blind interpreter enumeration, not the 4B's decompose-compose guidance, so a positive result would not support C21's specific claim (explorer = the 4B's tool-search). C12 itself found guided~=brute at depth-3 (~0.40) and concluded the crack is 'composition-structure + interpreter, not the model's planning.' Seed ONLY from model-GUIDED solutions (or run guided-only vs brute-only arms and report separately).
- TRAINING-RECIPE DRIFT vs C21: banked_tool differs from C21's banked1 by the ADDED depth-3 seeds only if LoRA rank(32)/epochs(3)/lr(2e-4)/alpha, the depth-1+2 pair set, and k_visible/m_hidden(8/8) are held identical. Frontier build_tasks used k_visible=10, which would shift the prompt distribution vs the 8-visible eval/C21 prompts. Any delta besides the depth-3 seeds confounds the depth-3 gain.
- THINK-MODE EVAL lets the banked model explore at test time; it is symmetric across base vs banked so it is fair for the C21-matched metric, but it means 'depth-3 coverage' includes test-time search, not pure installed capability. The deployable single-shot no-think greedy@1 should also be reported to show the capability was actually banked into the weights (C12/C21's deployable framing).

## Leakage risks
- TRAIN/EVAL FILE-WIRING LEAK: eval_ladder excludes only data/train_tasks.jsonl (untagged), keyed on (depth,target_ops). harvest.py writes tagged files (train_tasks{tag}.jsonl). If the tool harvest writes to any tagged/other filename, its depth-3 tasks are NOT excluded and can reappear verbatim as held-out eval tasks (direct memorization). The exclusion file must contain the UNION of every task set used in training (depth-1+2 self-harvest AND depth-3 tool harvest).
- BEHAVIORAL (not string) DISJOINTNESS GAP: disjointness is by exact (depth, tuple(target_ops)); a depth-3 eval task behaviorally identical to a harvested train task via commuting ops or an alternate decomposition (especially likely once the search emits different target_ops than the generator) passes the filter yet is effectively memorized. Dedup eval-vs-train by the full visible+hidden I/O behavioral signature, not by target_ops.
- UNFROZEN / UNPAIRED EVAL SET: eval_ladder regenerates eval_tasks.jsonl every run from the current train_keys and advances a seeded RNG past excluded tasks, so base (train_tasks = d1+2) and banked_tool (train_tasks = d1+2+3) runs get DIFFERENT held-out sets -> the base-vs-banked comparison is unpaired and the reported delta partly reflects different task draws. Generate the held-out set ONCE with the full exclusion and evaluate both arms on that frozen file.
- NOT A LEAK (verified, matches C21): hidden examples never enter the training text -- the search targets visible I/O only, prims_to_code renders a program (no labels), and the hidden set is used solely as an execution-oracle to filter which programs to bank, identical to C21's full_pass selection. Rendering search-found ops to reference_code is legitimately 'no teacher': the code is written by 4B-guided search + interpreter using the substrate's own primitive DSL, not an external model.

## Metric soundness
"The decisive metric matches C21 exactly: analyze.covK is think-mode coverage@16 via cov_at_k, and C21's base 0.00 -> banked 0.00 depth-3 was measured the same way, so the mode/estimator comparison is fair. The base=0.00 floor is genuine: wall_climbing make_task behaviorally rejects shallow-equivalents (min_depth_leq), so its depth-3 tasks are true-depth-3, unlike C12's flawed generator (40% collapsed). TWO soundness problems: (1) cov_at_k(c,16,16) returns 1.0 for ANY task with >=1 correct sample among 16, so covK = fraction of depth-3 tasks with at least one hit, and at n_per_depth=20 the pre-registered P2 bar '>0.05' means literally 'more than one task out of 20' -- inside binomial noise and clearable by a single lucky or memorized task. (2) It measures test-time-search coverage, not banked one-shot capability. Fix: raise n_per_depth substantially, pre-register a threshold above base's binomial upper CI, and additionally report deployable no-think greedy@1 as banked-capability evidence."

## Must-fix (all applied in prereg/scripts)
- Lock the decompose-compose search's primitive set AND parameter options EXACTLY to families.LIST_PRIMS (16 ops, families' PARAM ranges). Do NOT run the harvest with gen_tasks.PRIMS (23 ops)/decompose_lib.PARAM_OPTS. Assert the search can only emit ops/params that exist in the eval/generator DSL, and re-verify each harvested task is still behaviorally true-depth-3 under the 16-op DSL.
- Seed banking ONLY from model-GUIDED search solutions; exclude brute-force enumeration hits (run_search currently saves the guided+brute union). If you want the brute comparison, make it a separate arm and report it separately -- do not mix it into banked_tool.
- Guarantee full disjointness: (a) ensure the exclusion file eval_ladder reads (data/train_tasks.jsonl) contains the UNION of the depth-1+2 self-harvest tasks AND the depth-3 tool-harvest tasks; (b) add behavioral dedup -- drop any eval task whose visible+hidden I/O signature matches ANY training task, not just those with equal target_ops.
- Freeze one held-out eval task file (generated once with the full exclusion applied) and evaluate base and banked_tool on that identical file, so the comparison is paired. Do not let eval_ladder regenerate a fresh set per arm.
- Hold the training recipe identical to C21's banked1 (LoRA rank/alpha/dropout/epochs/lr, the exact depth-1+2 pair set, k_visible=8/m_hidden=8 prompts). The ONLY delta from banked1 must be the added guided-depth-3 seed pairs; render depth-3 prompts with the same 8-visible ident_prompt used in eval, not frontier's 10-visible _mono_prompt.
- Raise n_per_depth well above 20 and pre-register P2 as banked_tool depth-3 covK exceeding the binomial upper CI of base depth-3 (which is ~0), not a bare >0.05; a single solved task must not clear the bar. Also report per-task cov_full counts and how many DISTINCT depth-3 tasks were unlocked.
- Additionally report deployable no-think greedy@1 (and no-think coverage) at depth-3 for base vs banked_tool, to distinguish banked-into-weights capability from test-time think-mode search.
- Report the overlap statistics: number of distinct depth-3 behaviors harvested vs in the eval set, and confirm no eval depth-3 behavior is collapsible to <=2 under BOTH the 16-op and the search DSL, to rule out a spurious 'crack' via shallow-equivalents.
