# Design review — hypothesize-verify-wall (pre-GPU, 2026-07-08)

Three adversarial lenses (workflow); verdicts: confounds=sound_with_fixes, stats=sound_with_fixes, mechanism=sound_with_fixes

## Lens: confounds — sound_with_fixes

### Must fix

1. METRIC LEAK — lookup-table mimicry trivially passes the primary metric. `model_structure_correct` (experiments/qwen35_4b_crosssubstrate_structure/scripts/cross_substrate.py:89-96) compares model-program behavior on the 8 VISIBLE inputs against all fills of the true skeleton; the true fill reproduces the visible outputs by task construction, so ANY program that hardcodes the visible pairs (dict/if-chain) scores structure-correct with probability 1. The scaffold arm ('mentally execute... check... only then write the function') and the dsl_sft traces actively push a fit-the-examples fallback, so arms can 'win' the primary metric and fire the pre-registered C36-falsification rule with zero op-structure content. Decision (the design must state it): lookup-table output is a LEAK, not a win — it contains no op structure and fails hidden generalization. Fix: make the primary decision metric mimicry-robust — require that some true-skeleton fill matching visible behavior ALSO matches the model program's behavior on fresh probe inputs (hidden INPUTS and/or make_probes-style draws; labels never shown to the model, so the hidden-label boundary holds); report the legacy C36 metric alongside for continuity plus a per-arm hardcode/mimicry rate (visible-match but probe-divergent-or-error).

2. SFT-arm channel/format confound. The C45 recipe the design says it reuses trains with enable_thinking=False and the trace as assistant CONTENT ending in 'Answer: <digit>' (experiments/qwen35_4b_meta_induction/scripts/train_lora.py:35-37, gen_cot_general.py:36-37), and C45's own eval matched that channel (eval_induction.py:27). Deploying c45_zero and dsl_sft at 'K=12 think samples @1024' puts both adapters in a chat template and output channel they were never trained in, while eval extracts a ```python transform``` block — a flat result can be pure format/template noncompliance, not absent skill (exactly the lens-4 confound). Fix: pre-register per-arm channel alignment (either build dsl_sft traces into the think channel under the think template, or deploy SFT arms in their trained no-think channel at matched TOTAL token budget), report code-parse rate per arm×depth, and pre-register that a null with parse-rate far below base is 'format-confounded', not 'no transfer'.

3. Trace-generation policy unspecified and its oracle-purity unverifiable (C47 lens). The DSL hypothesis space is 256 (list) / 169 (string) skeletons at depth-2 and 4096 / 2197 at depth-3 with ~2 param fills per op — unlike C45's 4-candidate scan, exhaustive enumerate-and-verify cannot fit in a 1024-token trace, so gen_traces.py must implement a PRUNING policy. The README never specifies it. If pruning consults the true skeleton (candidate order, shortlist membership), the traces teach an oracle-guided shortcut ('first plausible guess is right') — an impure install that can neither transfer nor be interpreted. Fix: pre-register the policy as a pure function of visible I/O (truth-independent candidate order, failed verifications shown, drop tasks the blind policy cannot solve within the trace budget — C45's cot()->None pattern), verify at build time by regenerating each trace with the oracle blinded and byte-comparing, and report trace-length distribution vs the 1024 deploy budget.

4. Leakage spec is ambiguous and vacuous exactly where it matters. (a) 'op-composition dedup vs ALL eval tasks' must be pinned to op-TYPE-sequence (skeleton) level: if dedup is at (type,param) level, a training task sharing a depth-2 eval task's exact type-skeleton with different params counts as '0 leakage' while directly teaching the eval answer under a type-level success metric. (b) For depth-3 eval the dedup is vacuous (no depth-1/2 composition can equal a depth-3 one), and with ~1500 traces over 256/169 pairs, nearly every eval depth-3 skeleton will have BOTH of its contiguous depth-2 windows in training. Pre-register the stance: does a depth-3 lift on tasks whose sub-skeletons were trained falsify C36 ('cannot propose novel deep structure') or only show composition of banked shallow structure (C24/C28 territory)? Recommended: keep training as designed but stratify depth-3 results by window-overlap (both/one/zero windows trained) and make the C36-falsification wording conditional on the strata.

5. c45_zero regen-sanity gate missing. The adapter is regenerated from committed train_general.jsonl (original moved external per C43); if the regen is off (seed, data order, recipe drift), a flat c45_zero arm is an artifact of a failed rebuild, not evidence about transfer. Fix: pin the training seed and gate the arm on the regenerated adapter reproducing C45's headline (held-out a7 induce-via-generation ~1.00, cheap: 200 no-think episodes) BEFORE spending its DSL eval; hard-stop in run.py like the other gates.

### Should fix

## Lens: stats — sound_with_fixes

### Must fix

1. Decision-rule/power mismatch: at n=30 per cell with base at the C36 floor (~0), the paired-bootstrap CI-excludes-0 condition already requires an observed diff >= 0.133 (4/30), making the '+0.10' clause vacuous per-cell; power for a true +0.10 lift is 0.35 (80% power needs ~+0.18). Re-anchor the PRIMARY pre-registered contrast to pooled list+string at depth 3 (n=60 per arm: min significant diff 0.067, the +0.10 clause binds at 6/60, power 0.90 at true +0.15) and demote per-family-by-depth cells to characterization. Pooling is also the correct falsification grain for C36's universal 'across substrates' claim — a pooled lift falsifies it whichever family carries it; report per-family breakdowns descriptively to scope the result. (README.md lines 65-69, 74-76.)

2. Multiplicity guard against the C47 suggestive-cell trap: 'C36 falsified iff ANY arm's CI excludes 0 in any depth-3 cell' gives 6 uncorrected chances (3 arms x 2 families); if the true base rate at K=12 think sits slightly off floor (~0.05, plausible since C36 measured K=8), FWER is ~0.3. Pre-register: (a) Holm correction across the 3 pooled arm-vs-base depth-3 contrasts (or designate one primary arm, e.g. dsl_sft as the install ceiling), (b) one-sided direction of the test, (c) an explicit statement that secondary metrics (greedy@1, full-solve@K, mean structure-correct) and depth-2 cells can characterize but NEVER flip the falsified/scoped verdict.

3. Matched-compute accounting for 'beat sample-more': matched K=12 at output cap 1024 is not matched compute — the scaffold prompt adds input tokens and enumerate-and-verify CoT systematically consumes more of the 1024 budget than base CoT; c45_zero/dsl_sft use the shorter base prompt, confounding provenance with cost. Pre-register per-arm-per-cell reporting of mean prompt tokens, mean generated tokens/sample, total generated tokens/task, and forced-close rate (already promised), PLUS a compute-parity presentation: base re-evaluated at K' chosen to match scaffold's total generated tokens (base samples are short, so this is the cheapest arm to extend) or coverage-vs-cumulative-generated-tokens curves. State that the headline is claimed at matched K AND survives (or not) at matched generated tokens; note dsl_sft's one-time training cost separately per C43/C45 precedent.

### Should fix

- Bound the null verdict by MDE: 'C45 scoped iff no arm moves any cell' must be worded as 'no effect >= ~0.15 pooled (~0.18-0.20 per family)' — a true +0.10 lift reads flat ~44% of the time even pooled, ~65% per cell. The falsification direction is safely conservative; the scoping direction is the one at risk of over-claiming from an underpowered null.

- Install gate under-specified: pin the trace-format checker (reuse the exact programmatic template checker from gen_traces.py), pre-register the gate n (>=50 held-out depth-1 tasks; at n=30 a true-0.85 install fails a 0.8 point-estimate gate ~20% of the time), and state pass = point estimate >= threshold. Consider strengthening from format-reproduction to held-out depth-1/2 task accuracy — emitting the format is not evidence the enumerate-verify content is load-bearing.

- Add a missing third gate for c45_zero: require the regenerated C45 adapter to reproduce held-out a=7 induction on a cheap subset (e.g. >=0.8 vs C45's 0.905, n~50) before interpreting DSL transfer; single-seed regen variance otherwise makes a flat c45_zero uninterpretable (failed regen vs failed transfer).

- Trap gate specifics: pin the oracle-skelfill value-search budget R in the README (gate pass/fail depends on it; C36 used the skeleton_fill budget), state pass = >=26/30 point estimate per family-by-depth, and pre-register how oracle-failed tasks are handled in the full-solve secondary (exclude or keep).

- The arm-taxonomy readings ('scaffold lift without SFT lift = elicitation-sufficient'; 'SFT-only lift = installable-only') compare significance verdicts across arms — the significant-vs-nonsignificant fallacy. Either add direct paired arm-vs-arm bootstrap contrasts for any cross-arm claim or label the taxonomy explicitly descriptive.

- Pre-register the contingency where base depth-3 is NOT at floor at K=12 think (C36's 0.00-0.01 was K=8; think config may differ): the paired rule still applies but the vacuity/MDE arithmetic shifts; also fix decode seeds and lean on mean structure-correct per K alongside coverage@12 to tame the P(>=1 of 12) estimator's noise.

## Lens: mechanism — sound_with_fixes

### Must fix

1. Pre-register the scaffold prompt VERBATIM and the dsl_sft trace template/rulebook (checked-in prompts/trace-generator source, referenced from the README) before any full GPU run, and make both arms carry the SAME procedure. The scaffold text IS the arm's entire treatment; the README currently paraphrases it, leaving a free parameter that post-smoke tuning would contaminate — the pre-registered verdicts 'elicitation-sufficient' (scaffold lift w/o SFT lift) and 'installable-only' (SFT-only lift) are only interpretable if scaffold and traces teach the identical procedure delivered through two channels; otherwise the contrast confounds procedure identity with delivery channel.

2. The taught procedure must be FACTORIZED and evidence-pruned, not the whole-pipeline enumerate-and-verify currently sketched ('compose a candidate pipeline; mentally execute...'). Token arithmetic: one depth-3 whole-pipeline check on one list example costs ~60-120 think tokens (candidate ~10, three intermediate states of 4-8 ints ~15-25 each, verdict ~10) vs ~35/candidate in C45; budget 1024 minus preamble buys ~5-13 checks per sample against 4096 (list) / 2197 (string) skeletons — with K=12 that samples <0.5% of skeleton space, so a null is guaranteed-by-arithmetic and uninformative. Viable concrete procedures: (a) forward chaining — shortlist <=3 candidate FIRST ops from input->output features, apply the pick mentally to all visible inputs, recurse on the residual transform (~20-30 tokens/slot-check, ~9-12 slot-checks + one full verification pass fits comfortably in 1024); (b) output-feature backward anchoring of the LAST op (sorted output => sort last; all-non-negative => abs; shorter => take/drop/dedup) — noting destruction ops (depth_wall_anatomy: true-d2 k>=1 -> 0/8) stall pure backward chaining, so forward or hybrid is safer. Require the CPU-side trace generator to serve as the pre-GPU existence proof: depth-appropriate traces must fit within ~1024 think tokens, and pre-register RULEBOOK-RECALL (fraction of eval tasks whose true skeleton survives the deterministic feature->op pruning, computable oracle-side) as the diagnostic that decomposes any null into rulebook-failure vs model-failure.

3. Resolve the trace CHANNEL incoherence. The C45 recipe the design cites (meta_induction train_lora.py) trains the CoT as a no-think answer-channel target (enable_thinking=False, target = trace + 'Answer: ...'), but the design deploys dsl_sft 'think-mode' and evaluates ALL arms with matched K=12 think @1024 + 'Only a ```python code block' answers. Either (i) train traces into the think channel with the final code block as the answer (the bank_the_thoughts train_lora_think.py pattern — a recipe change vs C45 that must be stated), or (ii) train answer-channel traces and give dsl_sft/c45_zero a no-think eval seat with large answer_max (breaking the 'identical decode params' claim, which must then be re-scoped). As written, the install gate, the budget-1024 question, and the matched-decode claim don't compose; also note budget applies to the think channel only — final code rides in answer_max and does not eat the 1024.

4. Condition the decision rule on prior art the README omits: (a) C28 (bank_the_thoughts: depth-3 plan-trace SFT already lifts held-out depth-3 cov@16 base~0 -> 0.325, content-causal) and C33/C35 (banked depth-3 structure-cov 0.51) ALREADY broke the 'wall is un-installable' reading for installs fed oracle depth-3 CONTENT; (b) C21 (wall_climbing: depth-1/2 answer-banking -> depth-3 0.00->0.00) is the direct negative prior for cross-depth transfer; (c) C26 (thinking_lookahead: untaught thinking at budgets 1024 AND 2048 flat at step-1; traces show 'confused meta-reasoning, not enumerate-and-test' — internal brute-force refuted) is the closest negative prior for the scaffold arm. The design's actual novelty is exactly two deltas — scaffold = TAUGHT procedure with zero training (delta vs C26), dsl_sft = procedure traces WITHOUT depth-3 content (delta vs C28, C21-for-procedures) — and the depth-1/2-only training restriction is the load-bearing property that makes a depth-3 dsl_sft lift novel rather than a C28 replication. Add these to Prior anchors and state in the decision rule that 'C36 falsified' means falsified for content-free installs/elicitation; without this a positive gets misattributed and a flat result gets over-read as 'C36 hardens' when it may only replicate C21 depth-locality (the pre-registered 'depth-2-only = partial procedure deficit' cell partially covers this — say so).

### Should fix

- Trace well-posedness details to pre-register: traces must include genuinely FAILED candidate checks (not only the winner), must terminate via verification rather than assertion, and shortlists must be computable from visible I/O by the fixed rulebook alone — the true pipeline may be used only to guarantee termination (C45-standard); never adaptively widen a shortlist per-task until it contains the truth, or the model learns oracle labels dressed as search.

- Budget contingency: pre-register a forced-close threshold (e.g., >30% forced on scaffold/dsl_sft arms) that triggers a 2048-budget probe on a task subset. The README's comfort citation is weak: C47's '99% truncation still worked' is a JUDGE-seat result — a truncated verify still emits P(True), but a truncated SEARCH loop leaves no hypothesis for the answer channel to compile, so C44's serial-compute law bites harder in the searcher seat.

- c45_zero arm: add a regen-sanity gate (regenerated adapter must reproduce C45's heldfam_a7 induction >= ~0.85 on the committed set) — currently this arm has no gate and a dead adapter would silently read as 'no transfer'. Pre-register the asymmetric reading: positive = genuinely novel cross-substrate strategy transfer (nothing in prior art shows it); null = uninformative by format-mismatch construction (C43 OOF 0.30 and C44 OOF 0.13 failed WITHIN-format, so cross-substrate zero-shot has near-zero prior). Keeping it costs ~2.5-3h of a 9-11h day for a one-sided lottery ticket plus narrative closure of the C45 bridge — defensible, but designate it the first arm to cut on overrun, and never narrate its null as 'transfer failed'.

- Add CPU-side procedure-compliance metrics: fraction of scaffold-arm think traces containing candidate-check structure (op names + intermediate state + verdict), and dsl_sft trace-format adherence at depth-3 EVAL (the install gate only checks held-out depth-1) — without these, a null is ambiguous between 'procedure ignored' and 'procedure ineffective'.

- Consider a matched-dose answer-only SFT control (same depth-1/2 tasks, targets = reference code, no trace) so a dsl_sft depth-2 lift is attributable to the TRACE rather than task exposure; C47's flat matched-dose C18 rerun partially covers this but at ~20x lower dose (n=70 vs ~1500) on the 23-prim family. If the day cannot absorb ~2h more, pre-register the attribution caveat instead.

- Base-arm regime note: C36's wall numbers (0.00/0.01) were NO-think K=8; the new base is think@1024 K=12, so 'base replicates the wall' is an assumption, not a given (C9: thinking moves things). Internal comparisons stay valid either way, but measure it, and additionally report whether base CoT shows SPONTANEOUS hypothesize-verify structure — directly relevant to the elicitation verdict.

### Notes

Verdict rationale: the core design is well-targeted and its two novel cells are real — no prior experiment tests a TAUGHT serial strategy at the wall (verified: C26 tested untaught thinking, C25/C34 logit-level guidance, C20 steering, C21 answer-banking, C28 oracle-plan content) — but four coherence gaps would undermine the pre-registered verdicts if unfixed: unpinned scaffold text, an arithmetic-infeasible loop shape at budget 1024, a train/eval channel contradiction inherited from the C45 recipe, and a decision rule not conditioned on C21/C26/C28/C33. All four are cheap (CPU/wording) fixes with ~zero GPU cost. Answers to the five lens questions: (1) full enumeration of 4096/2197 skeletons in 1024 tokens is impossible by ~2 orders of magnitude; the prompt must teach a factorized, evidence-pruned chain (forward per-slot shortlisting, or output-feature last-op anchoring, <=3 candidates/slot) and yes, it must be pre-registered verbatim — it IS the treatment. depth_wall_anatomy bounds each half: propose-half runs at ~2x chance beyond the first primitive (the thing the scaffold must beat), verify-half is strong (plan-given execution 0.90-1.00; C32 oracle-skelfill 1.0, replicated 1.0 in cs_list/cs_string, so the >=0.85 trap gate will pass). (2) A good trace = deterministic feature->op rulebook pruning -> small candidate product -> mental execution with FAILED checks shown -> verification on a held-back example -> code; the true pipeline may steer trace construction (C45-standard, prompt-side clean since eval uses the base ident_prompt), but the pruning must be label-free and its recall reported; the trace generator doubles as the budget-feasibility existence proof. (3) C44 implies 1024 is marginal: ~60-120 tokens per whole-pipeline depth-3 check (vs C45's ~35 with 4 candidates) => only ~5-13 checks/sample; sufficient ONLY for a factorized procedure (~20-30 tokens/slot-check => ~350-500 tokens total); pre-register a forced-close-triggered 2048 probe; C47's truncation-tolerance is judge-seat, not searcher-seat. (4) c45_zero null is uninformative by construction (format mismatch; C43/C44 OOF failed within-format), and doubly so given the channel mismatch (C45 adapter emits answer-channel traces, eval is think-mode); keep only with a regen-sanity gate and asymmetric pre-registered reading — its value is a one-sided shot at a genuinely novel cross-substrate transfer result; it is the right cut on overrun. (5) Priors to cite: wall_climbing C21 (depth-locality of shallow-trained installs — the exact failure mode a dsl_sft depth-3 null would replicate), depth_wall_anatomy ('reliable compiler starved of hypothesis search' — the mechanism claim this experiment directly tests; 30x-per-op odds decay gives the quantitative base expectation: ~(2x chance)^3 ~ 1/512 per blind sample => a few percent skeleton-coverage at K=12 from the base prior alone), C26 (thinking alone doesn't breach; the scaffold's delta), C28+C33/C35 (trace/plan installs WITH depth-3 content already reach 0.325 cov / 0.51 structure-cov — the bound that makes depth-1/2-only training the load-bearing design choice). Minor confirmations: train_lora.py max_length 2048 accommodates factorized traces; eval-side skeleton metric (model_structure_correct) is behavioral and arm-agnostic; n=30/K=12 power limits and the +0.10 threshold are already pre-acknowledged and internally consistent (only large effects trigger).
