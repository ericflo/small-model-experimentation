# Adversarial design review (workflow) — verdict: sound_with_fixes

## dpo_vs_more_training
SFT_2x is necessary but NOT sufficient. It controls only for gradient steps on positives; it does not control for the objective shape or for exposure to negatives, so a DPO>SFT_2x result still confounds "the discrimination/negative signal" with "the contrastive-loss geometry / length effects." Fixes: (a) Match optimizer STEPS exactly, not epochs. Pair-count != positive-count, so "2x epochs" won't equal the DPO phase's step count; set the compute control to SFT-on-chosen with steps == (SFT steps + DPO steps) and report the counts. (b) Add a SHUFFLED/MISMATCHED-NEGATIVE DPO arm: identical objective, same chosen, but rejecteds re-paired across tasks so the same-task correct-vs-wrong signal is destroyed while the contrastive geometry, negative exposure, and length statistics are preserved. Real-DPO must beat BOTH SFT_2x (compute) and shuffled-DPO (objective shape) to attribute the gain to discrimination — the prior constrained_coverage_dpo run showed the shuffled control was load-bearing (shuffling hurt pass@1). "SFT on chosen+rejected-as-positives" is NOT a control, it is a diagnostic: it teaches wrong code and should HURT; run it only to confirm the negative's sign carries information, never as the compute baseline. Also measure the pre-training implicit-reward accuracy on the held pairs (fraction with logp_SFT(chosen) > logp_SFT(rejected)) before any DPO — that is the discriminability the DPO gradient starts from and it ties directly to C13's measured 2AFC ~0.73. Proof standard: real-DPO > {SFT_2x, shuffled-DPO} on a paired McNemar over the fixed 80 tasks, replicated on >=2 training seeds.

## what_greedy_gain_means
A greedy@1 gain does NOT by itself mean the model learned to discriminate/select correct-over-wrong. Two mechanisms produce the same greedy@1 rise: (a) genuine selection — a correct solution already in the k=16 support is re-ranked to the mode, tail preserved; (b) mode-collapse/distribution-sharpening — DPO lowers entropy and concentrates mass on fewer modes; if those modes are more-often-correct, greedy@1 rises while the tail (coverage@16) and diversity fall. Must report ALONGSIDE greedy@1, for every arm: coverage@16, functional diversity (distinct hidden-output signatures over the 16), distinct-code/AST count, mean pairwise edit distance or self-BLEU among the 16, sampling entropy on held-out prompts, and per-arm PARSE RATE. Pre-declare the decision rule: discrimination is SUPPORTED iff greedy@1 rises toward coverage@16 (the gap closes) WHILE coverage@16 stays within noise and diversity is roughly held — i.e., re-ranking within existing support. Sharpening is indicated iff greedy@1 up but coverage@16 AND diversity DOWN. Decisive per-task fingerprint: of tasks newly-correct-at-greedy under DPO, what fraction were already in the SFT model's coverage@16 support? Selection => essentially all are promotions from support. The honest structural prediction that strengthens the read: DPO trains only on tasks that already have a verified-correct sample (tasks with no correct sample in k=16 contribute no pair), so DPO has NO signal to cover a truly-uncovered task — a flat coverage@16 with a rising greedy@1 is the exact fingerprint of within-support selection, whereas a coverage@16 that DROPS is sharpening, not discrimination.

## negatives_and_reference
Same-task, on-policy negatives are the correct PRIMARY: fixing the prompt makes correct-vs-wrong code the only difference (isolates the signal), and downweighting the model's own errors directly targets the greedy@1 deployment distribution. Three caveats to guard. (1) The negatives come from banked_1280 while the DPO policy is SFT-from-base, so they are mildly OFF-policy w.r.t. the trained policy; ideally re-harvest negatives from the SFT arm itself (banked_1280 ~= that arm, so acceptable, but flag it). (2) Raw verified-wrong samples are dominated by EASY negatives (wrong primitive, parse/exec crashes); DPO on easy negatives teaches "emit runnable code," not depth-3 discrimination. Stratify and report all-negatives vs HARD near-misses (visible-correct-but-hidden-wrong, or shares the first-k correct ops, or minimal edit distance to a chosen solution); keep same-task as headline, near-miss as robustness. (3) Characterize chosen vs rejected LENGTH distributions and add a length-only baseline (does "pick longer/shorter" predict correct?) — otherwise DPO can fake a gain via a length heuristic. Reference: cached-reference anchored to the FROZEN SFT model is more robust here, because the reward becomes exactly "beyond SFT," the KL leash to SFT protects coverage@16 from collapse, and it makes the headline quantity precisely what is tested. WARNING grounded in the repo: train_constrained_dpo.py computes the reference via model.disable_adapter(), which returns BASE; for a continue-SFT arm that anchors to base and lets the policy drift off SFT — you must merge the SFT adapter into base before disable_adapter, or cache ref logps under the frozen SFT model. Reference-free (ORPO/SLiC) drops the ref pass (cheaper, less memory) but has no explicit KL leash so it is MORE prone to coverage collapse; ORPO partly compensates via its built-in NLL(chosen) term. The logp_rejected->-inf degeneracy is worst in vanilla reference DPO WITHOUT an NLL anchor; the guard is an NLL(chosen) anchor (ORPO bakes it in; cached-ref DPO must add it explicitly, as constrained_coverage_dpo did with nll_weight=0.08). Recommend cached-ref-DPO(SFT)+NLL-anchor as primary and ORPO as a robustness cross-check; report both so the conclusion is not an artifact of the reference choice.

## dpo_implementation
Grounded in the repo's train_constrained_dpo.py. LENGTH NORMALIZATION: response_logps(normalize=True) divides summed token-logp by token count = IPO/length-normalized DPO, not vanilla DPO (which sums). Decide deliberately and keep IDENTICAL between ref-cache and policy forward. Length-normalized is safer (kills length bias, damps the ->-inf blowup) but rescales the gradient so beta must be retuned; if unnormalized, first confirm chosen/rejected lengths match or you will learn a length heuristic. REFERENCE CACHING CORRECTNESS: cache ref logps under the frozen SFT model with byte-identical prompt template, tokenization, EOS handling, truncation, and completion mask as the policy; any mismatch silently corrupts the margin. Prefer on-the-fly ref via a frozen/merged copy over a separately-loaded adapter to guarantee identical tokenization, and ensure it yields SFT (merge SFT first) not raw base. COMPLETION-ONLY MASKING: score only completion tokens with the prompt masked; verify the shift-by-1 label alignment and that response_mask starts at len(prompt)-1 (repo convention) — an off-by-one leaks a prompt token or drops the first completion token. REGULARIZER: keep the NLL(chosen) anchor (repo nll_weight=0.08) to prevent the joint chosen+rejected logp collapse and preserve parseability (greedy@1 is parse-sensitive); optionally the ref-drift L2 anchor (anchor_weight=0.08). BETA: 0.05-0.1 (repo 0.05); too high freezes at SFT and manufactures a false null, too low collapses coverage; sweep {0.05,0.1,0.3}. LR/STEPS: LoRA-DPO overfits fast on few pairs — LR 1e-5..5e-5 (repo 3e-5), short and EARLY-STOPPED (repo max_steps=10, grad_accum=4, r=16/alpha=32, grad clip 1.0). MONITORING: log reward_margin AND absolute pi_chosen; the classic pathology is reward_margin growing while pi_chosen FALLS (both-go-down) => parseability decays; early-stop on a dev coverage@16 slice at the first sign coverage drops (sharpening onset). DEFAULTS: length-normalized cached-ref DPO(SFT), beta 0.05, LR 2e-5, NLL anchor 0.05-0.1, ~10-40 steps early-stopped on dev coverage, r=16/alpha=32, >=2 seeds; pin greedy (temp 0) for the headline and an identical temp/top-p/seed config for coverage@16 across all arms.

## Confounds
- Min-depth audit: the frozen n=80 depth-3 held-out is ~30-40% behaviorally depth<=2 (C12 retro-audit); a greedy@1 gain concentrated on collapsed/really-depth-2 tasks is not the frontier claim.
- Harvest/eval leakage: harvest tasks must be behaviorally deduped (function-signature AND op-composition) vs the n=80, not just task-id-disjoint.
- Compute mismatch: SFT_2x (epochs) does not equalize optimizer steps with the DPO phase (pairs != positives); match steps.
- Easy-negative confound: verified-wrong samples dominated by parse/exec crashes teach runnability, not depth-3 discrimination.
- Length confound: systematic chosen/rejected length imbalance lets DPO fake a gain via a length heuristic (esp. with unnormalized logp).
- Reference-anchor confound: repo disable_adapter() reference = base, not the SFT the arm continues from, giving the wrong KL leash.
- Off-policy negatives: negatives are from banked_1280, not the SFT-from-base DPO policy, weakening the 'downweight my own deployment errors' argument.
- Statistical power: n=80 with greedy@1 in 0.05-0.19 makes adjacent-arm CIs overlap (recurring arc limit); single training seed is insufficient.
- Null under-identification: DPO~=SFT has many benign causes (beta/LR, too few or too easy pairs, KL too tight), so a null does not prove 'can't discriminate' without a positive control and the directly-measured pre-train margin accuracy.
- Parse-rate confound: DPO can move greedy@1 purely by shifting parse rate up (or crash it down via collapse); track parse rate per arm.
- Construct slippage: DPO shifting the generation distribution is not the same construct as inference-time 2AFC discrimination (already weak in C13-C15); do not equate a DPO null with an inability to discriminate.

## Must-fix
- Add a shuffled/mismatched-negative DPO arm (same objective, rejecteds re-paired across tasks) in addition to SFT_2x; real-DPO must beat both to claim the same-task discrimination signal.
- Match optimizer STEPS (not epochs) between the compute control and the DPO phase, and report the step counts.
- Anchor the DPO reference to the FROZEN SFT model, not base: merge SFT before disable_adapter or cache ref logps under SFT, with byte-identical tokenization/masking verified between ref and policy.
- Report coverage@16 + functional diversity + distinct-code + sampling entropy + parse rate for every arm; pre-declare that discrimination requires greedy@1 up WITH coverage@16 preserved, and that coverage/diversity DOWN = sharpening.
- Per-task promotion check: fraction of DPO-newly-greedy-correct tasks already in the SFT model's coverage@16 support (selection) vs newly covered.
- Slice the eval by min-depth-audited TRUE depth-3 and report greedy@1 on the true-d3 subset separately; the collapsed subset is not the frontier claim.
- Behaviorally dedup harvest vs the n=80 held-out (function-signature AND op-composition), not task-id only.
- Characterize negatives (crash/parse-fail rate vs hard near-miss rate) and run all-negatives vs hard-neg-only DPO; report chosen/rejected length distributions and a length-only baseline.
- Measure pre-training implicit-reward accuracy (logp_SFT(chosen) > logp_SFT(rejected)) on the held pairs and connect it to the C13 2AFC ~0.73.
- Add a positive control that DPO CAN move greedy@1 on an easy-signal pair set, so a DPO~=SFT null is not under-identified.
- >=2 training seeds, paired McNemar across the fixed 80 tasks, pre-registered effect size given n=80 power; keep the NLL(chosen) anchor and early-stop on dev coverage@16, monitoring absolute pi_chosen for the both-go-down collapse; run ORPO as a reference-free robustness cross-check.
