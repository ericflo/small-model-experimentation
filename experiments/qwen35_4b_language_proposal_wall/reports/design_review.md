# Adversarial design review (workflow) — verdict: sound_with_fixes

## is_this_proposal
Mostly yes, with one branching-factor caveat and one missing control. At the abstract level it IS a fair proposal/induction analog of C32: both ask the model to infer a depth-D operator SEQUENCE from I/O examples, and both then "apply" it. It is arguably a CLEANER proposal probe than C32, because C32 could never actually measure proposal accuracy directly — asking the model to emit the DSL op-sequence solved 0.00 even at depth-1 (a pure FORMAT handicap), forcing C32 to pivot to a behavioral structure-proxy. In linguistic think-mode the model will naturally verbalize "the rule is gorm then vor then kel," giving a DIRECT proposal-accuracy read C32 never obtained — exploit this.

Does a failure ISOLATE induction? Only if two things hold, and the design as written does not guarantee either. (a) Application/simulation must be near-ceiling ON THIS SUBSTRATE. C37's "application is easy" was measured on successor CHAINS (single next-pointer per step). This task is harder application: at each of D steps you must select the correct one of R full 10-entry bijections and look it up. You cannot assume C37 transfers — you must add an application-only control (hidden rule GIVEN explicitly, same KB/query/rendering) and show it is near-ceiling per depth. Without it, a depth-3 failure is confounded by application difficulty, and the "isolates induction" logic collapses. (b) Failures must be decoded into wrong-STRUCTURE vs right-structure-misapplied (in think mode, read the proposed rule directly). C32's whole force came from the structure-vs-value decomposition; you need the analog here or "failure = induction failure" is asserted, not shown.

Bundling INDUCE+APPLY is fine and intended, PROVIDED the two controls above subtract off the APPLY component. As designed the subtraction relies on a different-substrate prior (C37), which is the weak link.

## well_posed_and_nonmemorizable
Non-memorizable: yes — made-up entities/relations, contamination-clean (inherits C37's discipline), and query-start disjoint from example-starts forces rule-use over lookup (each example gives only start->final, not intermediates, so a new start is unanswerable without the induced sequence + KB). Good.

Well-posed / solvable: several real gaps. (1) MIN-DEPTH not verified. With random bijections over only ~10 entities, the depth-D composition can equal a shorter (<D) composition on all entities — the exact C13 pathology where 40% of nominal depth-3 tasks were shallower-equivalent. Nominal depth then != true depth. Must reject any hidden rule whose composed bijection equals any shorter composition over the KB. (2) n_ent=10 caps distinct non-query starts at 9. If uniqueness at D=4 (4^D=256 candidate rules) needs more than 9 distinct example starts, you CANNOT make it well-posed and must fall back to "all-consistent-rules-agree-on-query" — but that fallback tests something EASIER (any consistent rule, not the unique one) and can inflate scores. Enlarge the entity pool (e.g., 15-20) and report the fraction unique-pinned vs fallback-agree, stratified by depth. (3) A depth-1 FLOOR check is mandatory: C32's proposal task was 0.00 at depth-1 for format reasons; if linguistic induction also floors at depth-1 there is no "wall," just a comprehension floor, and the whole depth-3 story is void. Establish the full depth-accuracy SHAPE. (4) Leakage: exclude query starts that are fixed points of the hidden rule or whose answer coincides with an example answer; shuffle KB facts and example order; verify no positional/recency/alphabetical shortcut predicts the answer (C37-style shortcut-hardening).

Information-theoretically solvable at k~5: for D<=3 with R=4, yes (log_10(64)~2 pairs, aliasing pushes it up, ~5 is plausible). For D=4 with n_ent=10 it may be infeasible without the fallback. Provide a reference solver that confirms each instance is pinned/agrees at the k actually used, and report k-per-instance (do NOT fix k=5 blindly across depths — higher depth needs more, and varying k across depth is itself a confound to disclose).

## modality_confounds
The formal-DICT arm is code-mode-confounded (C37: dict rendering triggered a python-block dump, d1=0.03) — it is NOT a clean formal-proposal number. But do NOT make the arc's C32/C36 formal wall the PRIMARY comparison either: C32 measured proposal via a behavioral structure-PROXY (op-seq emission floored at 0.00 by format), so "linguistic apply-to-query accuracy" and "C32 behavioral structure-coverage" are different measurement instruments and not directly commensurable. Anchoring the headline dissociation on that cross-instrument comparison is unsound.

The clean primary is WITHIN this experiment, same KB/relations/renderings, differing ONLY in simulation (rule given) vs proposal (rule induced) vs application-control — a within-substrate simulation-vs-proposal dissociation. For the surface-modality contrast, the PRIMARY rendering should be linguistic-SYMBOLIC ("Kel's gorm is Vor") — which C37 showed is BOTH code-mode-free AND semantic-prior-free, and is exactly what this task naturally is. Keep formal-dict as a confounded SECONDARY and report its code-mode rate (fraction of responses emitting a code block) so it is interpretable, not load-bearing. Use C32/C36 only as external corroboration.

Think is needed — as a PRIMARY treatment, not secondary as in C37. Proposal is SEARCH (enumerate candidate sequences, test against examples); C32's finding is precisely that the model can't propose deep structure in a forward pass, so the two distinct questions "can it propose no-think?" vs "does scratchpad search crack it?" are both first-class. Two must-fixes: (a) C37's think was truncation-confounded at budget 1024 — size the budget to allow candidate enumeration+testing and report truncation rate; (b) beware that with R=4 (4^3=64 candidates) a think scratchpad can brute-force-enumerate ALL rules, so a think success would be "in-context search," not "proposal" — this is why R must be swept up (see must-fix). Also: think-mode uniquely lets you SCORE the proposed rule directly (the read C32 could never get) — instrument for it.

## metrics_and_prediction
Metrics as stated (accuracy vs depth, per rendering, no-think + think, baseline ~1/n_ent~0.1) are fine but insufficient to license the dissociation. Add: (i) application-only accuracy per depth (rule given) as the ceiling that "isolates induction"; (ii) direct proposal-accuracy from think-mode verbalized rule; (iii) failure decoding into wrong-structure vs right-structure-misapplied; (iv) fraction unique-pinned vs fallback-agree; (v) code-mode rate for the dict arm; (vi) k-per-instance and truncation rate. Note the guessing baseline may exceed 1/n_ent if the query-answer distribution is skewed — compute the empirical majority-answer baseline instead.

Distinguishing "proposal walls in language too" from "task is just hard/underspecified" — this is THE crux and needs a pre-registered operational definition. A depth-3 result counts as a genuine PROPOSAL WALL only if ALL hold: (a) depth-1 induction near-ceiling (no format/comprehension floor); (b) application-only control near-ceiling at depth-3 on the same substrate (rules out application failure); (c) instances verified pinned/agree by the reference solver at the k used (rules out impossibility); (d) failures are predominantly wrong-STRUCTURE, not misapplication; (e) not attributable to code-mode or truncation. If instead depth-1 floors, or the application control also drops at depth-3, the correct reading is "task hard/underspecified," NOT "induction walls." Pre-register this decision rule.

Dissociation logic: the strong claim is a within-experiment interaction — proposal collapses at a depth where simulation (same substrate, rule given) stays near-ceiling, contrasted with C37's intact linguistic simulation. That interaction is what dissociates "induction = universal limit" from "simulation = formal-specific." Running simulation and proposal on the IDENTICAL KB makes this a within-subject contrast far stronger than citing C37 across experiments.

Power: n=80-100/cell (C37 used 80) is ample. The relevant effects are large (C37 simulation d3=1.00 vs a hypothesized proposal d3~0.1-0.3); at n=100 the CI on a proportion is ~+/-0.09, easily resolving a 1.00-vs-0.30 gap. Power is NOT the risk — CONFOUNDS in the comparison are. To detect a graceful-vs-cliff SHAPE across depths 1-4 you want >=100/cell. Suggest R-sweep {4,8,12} x depth {1,2,3,4} x {linguistic-symbolic, formal-dict} x {no-think, think} x {induce, apply-control} at n=100; that is the informative design, and the R-sweep is what makes the C32 comparison fair.

## Confounds
- Branching factor R=4 is far smaller than C32's ~20-primitive DSL (64 vs ~10^4 depth-3 candidates) and small enough for a think scratchpad to brute-force-enumerate — a think success would be in-context SEARCH, not PROPOSAL, and a wall/no-wall result is not comparable to C32 without sweeping R up
- Application on this substrate (select-1-of-R full bijections per step, then compose) is HARDER than C37's single-next-pointer successor chains, so C37's 'application is easy' does NOT transfer; a depth-3 failure is confounded by application difficulty unless a same-substrate apply-only control is near-ceiling
- Nominal vs true depth: random bijections over ~10 entities can make a depth-D rule equal a shorter composition (the C13 40%-shallower pathology) — must min-depth-verify the hidden rule
- Depth-1 comprehension/format FLOOR: C32's proposal task was 0.00 at depth-1 for format reasons; if linguistic induction also floors at depth-1 there is no wall to interpret
- Think truncation (C37's think confounded at budget 1024) — proposal needs search room; undersized budget masks capability
- Formal-dict CODE-MODE (C37 d1=0.03): the dict arm dumps code instead of reasoning; not a clean formal number
- Uniqueness fallback ('all consistent rules agree on query') tests an EASIER problem than unique identification and can inflate scores; k varying across depth is a separate confound
- Hidden rules with REPEATED relations (gorm-gorm-gorm) reduce to simulation, not proposal — mixes the two constructs
- n_ent~10 caps distinct non-query example starts at 9, potentially making D=4 uniqueness infeasible and forcing the fallback
- Leakage: fixed-point/identity query starts, query-answer coinciding with an example answer, and unshuffled KB/example ordering can leak the rule or the answer; empirical majority-answer baseline may exceed 1/n_ent

## Must-fix
- Add a same-substrate APPLICATION-ONLY control (hidden rule stated explicitly; identical KB/query/rendering) and show it is near-ceiling per depth — this is what actually subtracts off the APPLY component and licenses 'failure isolates induction.' Do NOT rely on C37 (different substrate: successor chains vs composed bijections)
- Sweep R up (e.g., {4,8,12}) — R=4 gives only 64 depth-3 candidate rules: too small to be a fair analog of C32's ~20-primitive DSL and small enough to be brute-force-enumerated in a think scratchpad, which would confound PROPOSAL with in-context SEARCH
- MIN-DEPTH-verify every hidden rule: reject any whose composed bijection equals a shorter (<D) composition over the KB (C13 pathology). Also disallow/segregate rules with repeated relations
- Establish depth-1 induction is near-ceiling and report the full depth-accuracy SHAPE (cliff vs graceful) — a depth-1 floor (as in C32's format-handicapped proposal) would void any 'depth-3 wall' claim
- Make THINK a primary treatment with an adequately sized, non-truncating budget (C37's 1024 truncated); report truncation rate. Use think-mode to read the model's VERBALIZED proposed rule for a DIRECT proposal-accuracy score (the read C32 never got)
- Decode failures into wrong-STRUCTURE vs right-structure-misapplied (the C32 structure-vs-value analog); most errors should be wrong-structure if the wall is proposal
- Pre-register the WALL decision rule: depth-3 counts as a proposal wall ONLY if depth-1 is near-ceiling, the apply-control is near-ceiling at depth-3, instances are solver-verified pinned/agree at the k used, failures are predominantly wrong-structure, and it is not code-mode/truncation — else classify as 'task hard/underspecified'
- Make the PRIMARY comparison within-experiment (same KB): simulation vs proposal vs apply-control, with the linguistic-SYMBOLIC 'gorm' rendering primary (code-mode-free AND semantic-prior-free per C37); treat formal-dict as a confounded secondary and report its code-mode rate; use C32/C36 as external corroboration only, not the primary formal number
- Enlarge the entity pool (>10) so D=4 uniqueness is achievable; verify each instance with a reference solver at the k actually used; report fraction unique-pinned vs fallback-agree and k-per-instance, stratified by depth
- Shortcut-harden per C37 (shuffle KB + example order; exclude fixed-point/identity query starts and query-answers that coincide with example answers; verify no positional/recency/alphabetical predictor) and use the empirical majority-answer baseline, not just 1/n_ent

## How addressed
- MANDATORY APPLICATION-ONLY CONTROL added (rule GIVEN explicitly, same KB/query/rendering) -- the ceiling that isolates induction; a depth-1 induction floor is only a WALL if application-only is near-ceiling.
- MIN-DEPTH verification: reject hidden rules whose composed bijection equals any shorter composition on the KB (the C13 pathology).
- Entity pool enlarged (n_ent 16) for uniqueness headroom; report unique-pinned vs fallback-agree.
- PRIMARY rendering = linguistic-symbolic (made-up relations: code-mode-free + semantic-prior-free); formal-dict = confounded secondary.
- THINK is first-class (proposal = search); no-think + think both run; budget 2048; report truncation.
- Pre-registered decision rule: PROPOSAL WALL iff (a) application-only near-ceiling incl depth-1, (b) induction at/near chance, on the SAME substrate. Else 'task hard/application-confounded'.
