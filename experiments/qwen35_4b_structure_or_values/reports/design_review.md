# Adversarial design review (workflow) — verdict: sound_with_fixes

## opseq_format_handicap
The depth-1/2 op-seq control is NECESSARY but NOT SUFFICIENT as stated, and this is the single highest-risk hole given repo history: the experiment_log for THIS experiment records that the first proposal design "parse the model's generated next-op FAILED -- the model reasons verbosely and gets truncated," which is why they abandoned generation for letter-logit reads. You are re-introducing the exact modality that already failed by truncation. Depth-1/2 format success does NOT license depth-3, because format competence is length/budget-dependent: a 1-2 op sequence is trivially in-format; a 3-op sequence at higher think-budget is where truncation and drift bite. So matching on DEPTH is not matching on the relevant nuisance (sequence length / generation budget). MUST-FIX: (a) separate PARSE-SUCCESS from SOLVE — report unparseable rate per depth as its own number; if you fold "unparseable" into "wrong skeleton" you will undercount structural knowledge and falsely conclude STRUCTURE-fails. (b) Add a depth-3-length format-capacity probe: hand the model the TRUE skeleton and ask it to transcribe it in the op-seq format (pure format, no reasoning) — measures format capacity AT depth-3 length, which the depth-1/2 solve control cannot. (c) Bound the format tax exactly as proposed by comparing op-seq coverage to the model's NATIVE Python coverage at matched depth (the whole arc — C11, banking, M4 — evaluated via Python), and report skeletonfill on the Python-derived skeleton too (parse the generated Python into an op-type sequence) so the structure signal does not hinge on a modality the model was never trained to emit. (d) Control think-budget and demo the format with depth-matched few-shot, but note the demos then risk leaking structure — see confounds.

## fill_search_power
The two controls are the right pair but the HEADLINE as written ("skeletonfill >> direct => wall is VALUES") is a FALSE DICHOTOMY, and the value-search being strong is exactly what breaks it. skeletonfill can exceed direct for a reason that has nothing to do with value-binding: a strong 216-combo enumerator over a permissive 8-visible filter will fill MANY skeletons — including wrong ones — to pass visible. So "skeletonfill >> direct" is consistent with "search is strong, model structure irrelevant." The decisive quantity is NOT skeletonfill vs direct; it is MODEL-skeletonfill vs RANDOM-skeletonfill, both on HIDDEN-generalizing solves, on TRUE-depth-3 tasks. The pattern that proves the wall is VALUES (model knows structure): model-skeletonfill-hidden >> random-skeletonfill-hidden AND >> direct, with oracle-skeletonfill ~1.0. The pattern that proves it is SEARCH (model does not know d3 structure): model-skeletonfill-hidden ~= random-skeletonfill-hidden despite both >> direct. Priors predict the second: C19/C30 found op-TYPE decodability is a "thread" at depth-3 (0.23, near chance), so the model likely does NOT know d3 structure and the headline would be wrong. On oracle-skeletonfill as ceiling: I verified PARAM_OPTS in decompose_lib.py EXACTLY equals the generator's _sample_param support, so oracle-skeletonfill will be ~1.0 largely BY CONSTRUCTION (the enumerator is handed the exact discrete param support) — it confirms the enumerator is correct, not a deep fact about searchability; do not oversell it. Report visible-pass vs hidden-pass as overfit rate = (visible-pass & hidden-fail)/visible-pass for EVERY arm (model, random, oracle); this is the C17 "overfit trap" (visible-pass/hidden-fail false-deploy) and is the honesty metric. MUST-FIX on matched-budget: "matched budget" must match the SEARCH-SPACE SIZE, i.e. random skeletons matched on depth AND number of parameterized-op slots (a random skeleton with more param slots gets more overfit chances at equal call budget); also add a shuffled-model-skeleton control (same op-type multiset, permuted order) to test whether the model gets ORDER right vs merely the SET.

## skeleton_nonuniqueness
Behavioral skeletonfill-coverage as primary with exact-type-match as secondary is CORRECT — I confirmed make_task only applies non-degeneracy checks (not identity, outputs vary, not all-empty) and does NOT reject alternative same-depth skeletons, so multiple op-TYPE sequences produce identical I/O and exact-match would systematically UNDERCOUNT structural knowledge (a valid alternative factorization scored "wrong"). Use exact-type-match only as a lower-bound sanity secondary. BUT the wrong-skeleton-fillable concern is not a footnote — it is the crux that determines whether behavioral skeletonfill is INFORMATIVE AT ALL. On a 16-op DSL with destructive ops (take/drop/filters), many behaviorally-equivalent factorizations exist, so I expect non-trivial random-fillability, especially on the ~40% of nominal-d3 tasks the arc's own min-depth audit found to be behaviorally depth<=2 (collapsed tasks have huge equivalence classes). Therefore behavioral skeletonfill is informative ONLY conditional on random-skeleton hidden-coverage being LOW, and ONLY on TRUE-depth-3 tasks. MUST-FIX: slice EVERY metric by verified min-depth (true-d3 vs collapsed) using an exact-BFS min-depth audit like the depth_wall_anatomy substrate — do not report on nominal depth; the arc has been burned by this exact confound repeatedly. Gate the "wall is values" claim on random-skeletonfill-hidden being low on true-d3; if random is also high, the correct finding is "the DSL is value-fungible / search is strong," a different and less mission-relevant conclusion than "model knows structure."

## metrics_and_controls
The decomposition is sound IN PRINCIPLE but has an implementation trap that muddies (skeletonfill - direct): skeletonfill MUST reuse the EXACT op-type sequence the model generated for direct-cov (strip its params, re-enumerate), NOT a fresh re-prompt for "just op-types." Only then does (skeletonfill - direct) isolate the pure VALUE contribution on IDENTICAL structure; a re-prompt changes the structural proposal and confounds value-binding with structure-variation. Second, skeletonfill - direct conflates value-binding with a serialize/decode failure (knows op-types, can't emit a coherent generated sequence); the shared-structure construction above resolves this since direct and skeletonfill then differ ONLY in params. On the DEPLOYABLE metric: the arc's north-star is greedy@1 forward-pass elicitation, and skeletonfill is NOT that — it is test-time execute-filter SEARCH over 216 combos with selection among visible-passers, i.e. exactly C17's "selection is free but is sample-more" and the C31 finding that the parameter is SURFACE-READABLE by a trivial I/O classifier (so "value search cracks it" is largely PREDICTED by C31, not new latent capability). Frame skeleton-then-fill honestly as a tool-augmented SEARCH recipe (decompose/C12 family, interpreter-as-executor per C17-free-selection), benchmarked against monolithic greedy@1/pass@k AND matched-budget random-skeleton (the brute bar the frontier experiment already adopted) — NOT as a forward-pass gain. Report a genuine deployable single-shot number: model's GREEDY (k=1) op-seq skeleton + value search, so the "one structural guess + cheap search" recipe is quantified, plus whether banking the found solutions lifts monolithic greedy@1 (the M4 bound). POWER: base direct d3 ~0 and the arc's own audit shows true-d3 is ~60% of nominal with monolithic true-d3 = 0/24 and decompose 4/24 (17%); n=40/depth yields ~24 true-d3, on which a 17% rate is 4/24 (Wilson 95% CI ~7-37%) — badly underpowered for "skeletonfill >> direct" when both are single-digit. MUST-FIX: generate ~150-200 nominal d3 to net >=100 TRUE-d3 after min-depth filtering; pre-register the model-vs-random-skeletonfill-hidden contrast on true-d3 as PRIMARY; use paired McNemar (same tasks across arms) with Wilson CIs; also increase visible examples to >=10 (the frontier work bumped to 10 to constrain generalization; 8 visible + 6 hidden here is a REGRESSION that raises the overfit/false-positive rate on a 216-combo search).

## Confounds
- Few-shot format demos leak/bias structure: depth-matched demos give away the skeleton pattern and bias the op-type distribution toward demoed ops; needs a demo-ablation or neutral-menu control (cf C30 neutral-placebo).
- Nominal vs TRUE depth: make_task does not reject same-depth-equivalent or shallower-equivalent skeletons; the arc's own audit found 40% of nominal-d3 are behaviorally depth<=2 -- collapsed tasks inflate BOTH random-fillability and skeletonfill; must slice by exact-BFS min-depth and report true-d3.
- Truncation/verbose reasoning: the prior op-seq-generation design in THIS experiment failed by truncation; unparseable output will masquerade as structure-failure unless parse-success is reported separately and think-budget is controlled.
- skeletonfill must reuse the SAME generated op-type sequence as direct-cov (strip params) or (skeletonfill - direct) is muddied by structure variation, not pure value-binding.
- Matched-budget definition: random skeleton must match depth AND #parameterized-op slots (search-space size), else budget/search-breadth mismatch confounds the model-vs-random comparison.
- Fill-grid == generator support (verified in decompose_lib.PARAM_OPTS): oracle-skeletonfill ~1.0 is near-trivial by construction (enumerator handed exact discrete support), not evidence values are 'deeply' searchable.
- C17 overfit trap: selecting among 8-visible-passers false-deploys visible-pass/hidden-fail programs; must report abstention/overfit rate, not just coverage.
- C31 restatement risk: the parameter is surface-readable by a trivial I/O classifier, so 'value search cracks depth-3' is largely predicted by C30/C31; the ONLY new mission-relevant signal is model-skeleton > random-skeleton on true-d3 (structure is the latent part), which C19/C30 predict is near-chance at d3 -- the design must be powered to detect that NULL and not foreclose it.
- Selection channel = sample-more: skeletonfill is test-time search, not greedy@1 elicitation; must bench vs monolithic greedy@1 and matched-budget brute (random), per C17.

If model ~= random, the finding is 'search is strong / DSL is value-fungible,' not a value-binding wall.

- Report visible-vs-hidden overfit rate (visible-pass & hidden-fail)/visible-pass for model, random, and oracle arms as a headline honesty metric (C17 false-deploy trap).
- Power up: generate ~150-200 nominal d3 to net >=100 true-d3; raise visible examples to >=10 (8 is a regression vs the frontier's 10 and inflates false positives on a 216-combo search); use paired McNemar + Wilson CIs; pre-register the model-vs-random true-d3 contrast as primary.

- The decisive test per review = model-skeleton vs random-skeleton on MIN-DEPTH-verified true-d3; C19/C25 predict the model does NOT know d3 structure (first-op is a thread 0.23).
