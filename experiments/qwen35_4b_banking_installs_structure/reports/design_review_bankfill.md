# Adversarial design review (workflow) — verdict: sound_with_fixes

## tautology_vs_new
The COVERAGE metric is near-tautological and should NOT be the headline. By construction, whenever C33's struct-cov=1 (some param-fill of the TRUE skeleton reproduces the model behavior B, per model_structure_correct in skeleton_fill.py), the true skeleton is necessarily in the enumerated 4096, and C32's oracle-skelfill=1.0 guarantees value-filling it against the true outputs finds a hidden-correct program. So bank+fill COVERAGE >= struct-cov 0.512 is pinned by construction; it can only rise via a wrong skeleton getting a lucky true-output fill, which is spurious inflation, not a real gain. Worse: the honest brute-force bar (enumerate all 4096, fill against true outputs, NO model) has coverage ~1.0 (true skeleton is always in 4096 and oracle-skelfill=1.0), so coverage is uninformative for both arms. What is genuinely NEW and worth running is the DEPLOY number under single-pick with a real visible-overfit selection problem, which BOTH struct-cov and oracle-skelfill sidestep. C33's claim that bank+fill 'would deploy at ~0.512' is an over-claim: 0.512 is a coverage ceiling assuming you pick the right candidate; the end-to-end run's actual contribution is to replace that inferred ~0.51 with the TRUE single-deploy number (expected strictly below 0.512), and to confirm that behavioral skeleton-inference recovers the structure without oracle even when 4096 candidates compete (report inference yield: fraction of samples whose B matches >=1 skeleton, and mean inferred-set size). To be non-tautological it must (a) report DEPLOY not coverage, (b) include the brute-force-all-4096 deploy bar, (c) the B-filter must be shown to be load-bearing (bank narrows 4096 -> few skeletons, cleaner pick) rather than redundant with search.

## skeleton_inference_correctness
Two distinct 'false-solve' cases must be separated. (1) A wrong op-type skeleton fills against true outputs and passes visible AND hidden: this is NOT a false solve for a behavioral task -- the program is hidden-correct on all 14 examples, so counting it is fine; min-depth verification (min_depth_leq on n_probe=6) and DSL non-fungibility keep it rare. It inflates COVERAGE but never produces a wrong DEPLOY. (2) The DANGEROUS case: a wrong skeleton passes all 8 VISIBLE (true outputs) but FAILS the 6 hidden -- a visible-overfit trap that, if selected, deploys wrong. C32's random R200=0.108 is the WRONG safeguard for this: it measures random structure passing HIDDEN, not the visible-overfit rate among visible-passers. The right safeguard is to DIRECTLY measure, per task, the fraction of search-generated visible-passers that fail hidden (C17's analog: 13/74 = ~18% overfit, 90.2% visible->hidden -- but that was for MODEL candidates; a 4096-skeleton search generates a far larger, more adversarial visible-passer pool, so expect a worse overfit rate). Enumerating 16^3=4096 with first-visible-input pruning is correct and tractable: it is exhaustive over DSL structure (true skeleton always included), and pruning is sound as long as you eliminate a skeleton only when NO param-combo (not a greedy per-op choice) matches B on input-1. Worst cost is 4096 x up to 216 param-combos (max grid 6, 6^3) x 8 inputs x 3 ops per sample; memoize apply_op(op,k,input) and most skeletons die on input-1, so it is seconds-to-minutes/task, pure CPU. One correctness gap to flag: the model's Python may use params OFF the OPTS grid (add_k range is only {-3..3}, mul_k {-2,2,3}) or non-primitive code, so B may be un-inferrable (S empty) for genuinely-correct programs -- measure and report inference yield so you do not silently undercount.

## deploy_metric
DEPLOY is the whole point and needs a principled, leakage-free selector -- 'first' is order-dependent and unacceptable as the primary. Because the candidate pool is a SEARCH over inferred skeletons (not model samples), C17's 90.2%-visible->hidden does NOT transfer; expect a materially larger deploy-vs-coverage gap. Primary selector should be output-consensus / self-consistency: generate M fresh random inputs (min-depth verification already guarantees fresh inputs separate true-depth-3 behavior from collapsed overfits), run every visible-passer on them, and deploy the program whose output-vector is the plurality mode -- true-skeleton fills are all behaviorally identical and dominate when correct, while overfit traps scatter. This uses only the free interpreter (C17/mission-legal, no hidden leakage). Report two secondary selectors to bound it: (b) model-vote -- deploy the skeleton proposed by the most of the k=8 model samples (uses banking's own structure signal as tiebreak); (c) first-visible-passer as a pessimistic floor. Given C17 (visible-passers ~90% hidden-correct for model candidates), a well-selected deploy could approach cov but the search pool will drag it down -- I would predict deploy landing between banked greedy 0.200 and struct-cov 0.512, likely near or modestly above cov@8 0.362, NOT at 0.512. Report deploy under all three selectors plus the directly-measured visible-overfit rate, so the coverage(>=0.512)-to-deploy gap is fully attributed to selection.

## baseline_and_budget
Comparing bank+fill to banked greedy@1 0.200 and cov@8 0.362 is right; comparing DEPLOY to the 0.512 structure ceiling as a target is WRONG framing -- 0.512 is a coverage ceiling (oracle/any-passer), not an achievable single-deploy, so 'recovers the +0.15 value tax' should be restated as 'how much of the value tax is recoverable UNDER single-deploy' (deploy in (0.20, 0.51]). The comparison that is currently MISSING and is the actual crux: a brute-force-fill-all-4096 DEPLOY bar (structure NOT from the model). Its coverage is ~1.0 trivially, so if its DEPLOY (under the same consensus selector) is close to bank+fill's deploy, banking added no deployable value -- search alone suffices; if brute-force deploy collapses because 4096 skeletons drown the pick in visible-overfit traps while bank's B-filter yields a clean few-skeleton set, THAT gap is banking's deployable lever (efficiency + pick-cleanliness). Include it. Budget is not a GPU concern: the k=8 generations are the only GPU cost and are essentially the same as C33's already-run cov@8 pass; the 4096-skeleton search is pure CPU and cheap after memoization + first-input pruning. The real budget/plumbing snag: C33's runs/bank_banked.json stores only aggregated booleans, NOT per-sample code or behavior B, so you MUST either persist code from a fresh generation pass or re-run -- and if you regenerate, recompute struct-cov on the SAME new samples so deploy and its 0.512 anchor are on identical generations (do not compare a fresh deploy to the old 0.512). n=80 has CI ~0.105 (verdict.json struct_cov_ci); a deploy delta over greedy 0.200 or cov 0.362 must clear that, so pool the 2nd banking seed or extend to n>=160.

## Confounds
- Coverage is pinned >=struct-cov (0.512) by construction and the brute-force-all-4096 bar has coverage ~1.0 -- so coverage is uninformative for both arms; only DEPLOY discriminates.
- C33 runs store only aggregated booleans, not per-sample code/B; regenerating k=8 introduces fresh sampling variance -- recompute struct-cov on the SAME regenerated samples, do not anchor a new deploy to the old 0.512.
- Model may emit params OFF the OPTS grid (add_k only {-3..3}, mul_k {-2,2,3}) or non-DSL code, making B un-inferrable (S empty) for correct programs -> silent undercount; report inference yield.
- 8-input B under-determines the skeleton (behavioral aliasing on 8 points) -> inferred set S is a superset of true structure, inflating candidate count and overfit-trap exposure; consider inferring S from more probe inputs than the 8 visible.
- The B-filter is load-bearing: without it the pipeline is brute-force structure search (C32 territory), not banking -- must show bank-filtered achieves near-brute-force deploy with a tiny skeleton set.
- Only 6 hidden examples grade deploy correctness (vs C17's 8) -- weaker overfit detection; the consensus selector on fresh inputs partly compensates but the hidden-pass proxy is slightly looser.
- Selection must be leakage-free: the deploy selector may use only visible examples + fresh self-generated probe inputs, never the 6 hidden.

## Must-fix
- Make DEPLOY (single-pick) the headline metric and demote COVERAGE -- coverage is pinned >=0.512 by construction (true skeleton always in 4096 + oracle-skelfill=1.0) and is ~1.0 for the brute-force bar, so it proves nothing new.
- Add the brute-force-fill-all-4096 DEPLOY arm (structure NOT from model) under the identical selector; banking's deployable value is precisely bank-filtered-deploy MINUS brute-force-deploy, plus the skeleton-set-size reduction. Without this arm the run cannot show banking added anything over free search.
- Use a principled leakage-free selector (output-consensus on M fresh random inputs) as primary; report model-vote and first-passer as bounds. Do NOT use 'first' as primary.
- Directly measure the visible-overfit rate among search-generated visible-passers (fraction passing 8 visible but failing 6 hidden); do not rely on C32 R200=0.108, which measures the wrong quantity.
- Persist per-sample generated code + behavior B (patch bank_structure.py, which currently dumps only booleans); reuse or regenerate the k=8 samples and recompute struct-cov on the SAME samples so deploy and its 0.512 anchor are on identical generations.
- Report skeleton-inference yield: fraction of samples whose B matches >=1 of the 4096 skeletons and mean inferred-set size, to confirm behavioral recovery works without oracle and to catch off-grid-param undercounts.
- Reframe the claim from 'recovers the +0.15 tax to reach 0.512' to 'fraction of the value tax recoverable under single-deploy' (expected deploy strictly < 0.512); 0.512 is a coverage ceiling, not a deploy target.
- Verify first-input pruning is SOUND (eliminate a skeleton only if NO param-combo matches B[0], considering params jointly) and pool seed-2 or extend n>=160 given CI ~0.105 at n=80.
