# Adversarial design review (workflow)

**Verdict:** sound_with_fixes

## Upsampled control
MOSTLY CORRECT, matched on the axes that matter, with three fixes needed. What is matched: total examples (130 d1+2 + 640 d3 = 770 both), depth mixture (640/770 = 83% depth-3 both), epochs=3 -> identical optimizer step count, identical LR/cosine schedule/warmup (same #steps), and the d1+2 portion is C21's exact pairs held constant. Repeating 40 pairs to 640 EXAMPLES (then epochs=3) rather than keeping 40 pairs at epochs=48 is the BETTER match: same dataloader length, same shuffle statistics, same epoch boundaries, and the 16 copies get shuffled through each epoch (not back-to-back). So this is a genuine diversity-vs-compute isolation: both arms deliver 640*3=1920 depth-3 example-visits; up40 concentrates them on 40 functions (48 visits each), train_640 spreads over 640 functions (3 visits each). FLAWS/FIXES: (1) The 40 distinct MUST be exactly the N=40 dose's 40 pairs (the nested first-40), not a fresh random 40 — otherwise the contrast is confounded by WHICH 40 and you also lose the free bonus comparison (up40 vs the N=40 dose point isolates compute at fixed diversity, giving a clean 2x2). (2) Single training seed cannot support 'up40 ~= 640 -> compute' — a near-tie must survive >=2-3 seeds or it is indistinguishable from seed noise; this is the weakest point of the split. (3) Pre-register the decision rule with the CI: 'up40 lower-CI above train_640 upper-CI' vs 'non-overlapping below' vs 'overlapping (partial)'; a partial result (e.g. up40 0.28 vs 640 0.375) is the likely outcome and must be interpretable, not just the two extremes stated. Minor caveat (not a confound to remove): with only 40 distinct pairs, minibatches will sometimes contain duplicate examples, lowering effective per-step gradient rank — but that IS a consequence of low diversity, so it does not confound the diversity interpretation.

## C23 reuse validity
VALID, and the exclusion design is the correct mechanism, but it hinges on three checks that must be run and reported, not assumed. Why reuse is legitimate: the base model and the banked_640 adapter are bit-identical objects and the frozen held-out set is unchanged, so their eval numbers are literally the same numbers — no re-run needed; only the new 1280/2560 adapters need evaluation on that same frozen set. The 640-train's leakage was 0 and stays 0 because train_640 is untouched. Because the held-out is FROZEN (you cannot delete held-out items), 0-leakage for the extended arms must be enforced on the HARVEST side: any newly-harvested function (pairs 641..2560) whose func_sig OR op-composition collides with a held-out function must be dropped. This is load-bearing and gets HARDER as N grows: I measured that 640 depth-3 tasks already collapse to only 555 distinct behavioral functions over a 32-concrete-op space, so at N=2560 you are consuming a large fraction of the depth-3 function space and collisions with the 80 held-out functions become likely. MUST verify eval_ladder prints leak_fn=0 AND leak_op=0 for the 1280 and 2560 train_tasks. Second, nesting is only real if it is a strict superset: assert train_640's exact pair-set is a subset of 1280's, and 1280's of 2560's (hash the {prompt,code} pairs). tool_harvest is deterministic under seed 202 so the first 640 SHOULD reproduce, but rejection-sampling divergence must be checked, not trusted — otherwise a non-monotone blip is re-sampling, not saturation. Third and most important for the SATURATION reading: report DISTINCT func_sig count per dose. The harvest does NOT dedup among itself, so '2560 pairs' may be ~1000-1500 distinct functions; if the curve flattens because distinct-function supply flattens, that is a harvest-diversity ceiling masquerading as a representational saturation — the opposite of a capacity statement. A true saturation claim needs the curve plotted against DISTINCT functions banked, not raw pair count.

## Depth-4 design
Sound skeleton, but as stated it under-specifies the baseline and dose, and one guarantee needs an explicit assertion. (A) Scaffold MUST include the 640 depth-3 (plus 130 d1+2): the C23 recipe is 'install rung N-1, then add rung-N data', and C23 showed depth-3 became deployable at 640. Testing depth-4 from a depth-2 scaffold is a different, harder, non-analogous question and would confound a null ('recipe repeats' requires the installed d3 base). Include d3. (B) Do NOT test a single N — mirror the 40/160/640 dose ladder so you can make the same data-limited-vs-cap argument for depth-4; a single dose cannot distinguish 'installs' from 'installs a little'. (C) True-depth-4 guarantee: I verified it is actually SOUND for the list family, not just asserted. make_task rejects any task solvable in <=3 via min_depth_leq at cap=3; a full depth-3 BFS is <=33,824 merged states < the 60,000 seen_cap, so the check is COMPLETE (does not early-terminate) for 32 ops. And because the 6 probes are a subset of the 16 task examples, any depth-<=3 program the harvester's BFS (400k cap, all 16 examples) could find would also match the 6 probes and would already have been rejected by make_task — so harvested depth-4 solutions are guaranteed length-4, NOT relabeled depth-3. Still add two cheap asserts: (i) log that min_depth_leq's frontier exhausted (seen never hit 60k) for every accepted depth-4 held-out task, and (ii) report the harvested found-depth histogram and confirm all == 4. (D) Held-out cleanliness: eval_ladder already dedups vs ALL train_tasks by func_sig AND op-composition; confirm leak=0 vs the d1+2+3+4 union. (E) THE MISSING BASELINE: the install-attribution control is NOT the raw base — it is the d1+2+640d3 scaffold WITHOUT depth-4 data, evaluated on the depth-4 held-out. If that scaffold-only model is already >0 on depth-4 (depth-3 skill partially transferring), the 'depth-4 install' delta must be measured against it, not against ~0 base. (F) Add a depth-3 guardrail post-training (analogous to C23's depth-2 guardrail) to rule out catastrophic forgetting of the scaffold when d4 data is added.

## Confounds
- SATURATION (ARM1): harvest does not self-dedup; 640 pairs are only 555 distinct functions, so a flattening cov@16 at 1280/2560 may be a harvest-diversity ceiling (running out of distinct depth-3 functions), not a model saturation. Plot vs distinct-function count.
- SATURATION (ARM1): coverage@16 compresses toward its K=16 ceiling as per-sample pass rate rises; a plateau in cov@16 near 0.7-0.9 can be a metric ceiling. Track mean per-sample pass rate and greedy@1, not only cov@16.
- SATURATION (ARM1): n=80 held-out gives Wilson CI half-width ~0.1; detecting a subtle plateau (0.375 -> 0.45 -> 0.48) is underpowered at that n.
- ALL ARMS: single training seed. C23's own 'avoid' note says CIs are eval-noise only with no seed error bars. A plateau, a data-vs-compute tie, and a depth-4 delta can each be seed artifacts; >=2-3 training seeds needed at the decision points.
- NESTING (ARM1): if the 1280/2560 harvest is not a verified strict superset of the exact 640 pairs, non-monotonicity is re-sampling not capacity.
- LEAKAGE (ARM1/ARM3): collision probability of new harvest with the 80 frozen held-out functions grows as the finite depth-3/4 function space is consumed; without harvest-side func_sig+op-comp exclusion the extended arms silently leak.
- DATA-VS-COMPUTE (ARM2): if the 40 distinct pairs are not the nested N=40 subset, WHICH-40 confounds the diversity contrast.
- DEPTH-4 (ARM3): attributing a depth-4 delta to install requires the d1+2+640d3 scaffold-only baseline on the depth-4 held-out; raw-base ~0 is the wrong reference if depth-3 skill transfers.
- DEPTH-4 (ARM3): omitting the 640 d3 from the scaffold would confound a null result (wrong scaffold, not a hard rung).
