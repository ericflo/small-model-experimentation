# Adversarial design review (workflow) — verdict: sound_with_fixes

## does_it_isolate_retrieval_vs_induction
NO, not as built — the design has no cell that is simultaneously NOVEL and GENERALIZABLE, which is exactly the cell the retrieval-vs-induction question turns on. The 2x2 you actually need is {familiar, novel} x {generalizable-rule, arbitrary-map}. You have three familiar-generalizable ciphers (caesar, atbash, affine) and one novel-arbitrary (random). Two quadrants are empty. As you note, random-unseen ~ chance is TRIVIALLY true (no rule exists to induce OR retrieve) and is not evidence for retrieval; it only shows the model isn't hallucinating structure. So the entire discriminating signal must come from the GENERALIZABLE rules, and there the design fails a second way: affine is NOT a good novel rule. It is a named classical cipher (Wikipedia/CTF/crypto-tutorial-saturated); Qwen almost certainly retrieves "affine cipher" the same way it retrieves caesar/atbash. So caesar>atbash>affine, if observed, is entirely consistent with "all three retrieved, ranked by execution/eyeball-inducibility," with ZERO induction of anything novel. The claim "ICL = retrieval not induction" is therefore not testable with these four cells: you can never observe the induction-of-novel-generalizable-rule outcome because you never present one. Fix (mandatory): add >=1 genuinely novel-but-fully-generalizable cipher the model provably cannot retrieve but could in principle induce from the examples. Best option (also solves confound #2): CAESAR-IN-A-SCRAMBLED-ALPHABET — fix a random permutation P of the 26 letters, then encrypt by shift-by-k within P's ordering (i.e., cipher = P[(P^-1[c]+k) mod 26]). Same 1-parameter "shift by k" rule and same description length as caesar, but the alphabet order is arbitrary so it cannot be retrieved as a named skill; it is still 100% determined and generalizes to unseen letters IF the model induces the (revealed-by-examples) ordering+shift. The discriminating result is: standard-caesar-unseen HIGH while scrambled-caesar-unseen ~ chance, at matched rule-complexity = retrieval, not induction. If scrambled-caesar-unseen is also high, the model genuinely INDUCES novel generalizable rules and your headline flips. Also add a "rule-stated" control (give the rule in words, e.g. "shift each letter by 7") to separate can-execute-the-rule from can-recover-the-rule-from-examples.

## complexity_and_familiarity_matching
"Same bijection class" is the WRONG complexity control, and the code makes this vivid. Matched as OUTPUT bijections over 26 letters, but wildly UNMATCHED on every axis that governs few-shot success: (a) key/description entropy — caesar log2(23)~4.5 bits (rng.randint(3,23)), affine log2(5*25)~7 bits, random log2(26!)~88 bits, and atbash is 0 bits (make_cipher returns a SINGLE fixed cipher — there is literally one atbash, so it has no key variance at all while caesar/affine/random are distributions; atbash is a single point estimate, not comparable); (b) program/Kolmogorov length — "shift k" vs "reverse" vs "a*i+b" vs 26 explicit pairs; (c) inducibility-from-partial-examples — caesar preserves alphabet ADJACENCY (a->d,b->e...), so "output is input shifted N along abc..." is eyeball-inducible from 2-3 examples with NO retrieval; atbash is a reflection; affine needs multiplicative structure; random is impossible. So caesar could win purely because it is the SIMPLEST and most eyeball-inducible pattern, which is the OPPOSITE of the retrieval story (it would be easy INDUCTION). Description-length fully confounds familiarity: you cannot tell "caesar succeeds because FAMILIAR (retrieved)" from "caesar succeeds because SIMPLE/short-program (induced)." Disentangle with the scrambled-alphabet caesar: it holds description-length (1 param, shift-by-k) and inducibility-structure fixed while removing familiarity and removing adjacency-preservation. Pair it with standard caesar. That single pair is the crux experiment and isolates pure familiarity. Also matched-complexity should be operationalized and MEASURED, not asserted: run a zero-shot familiarity probe per cipher (e.g., "encrypt HELLO with a caesar/atbash/affine cipher") to get this model's actual familiarity ranking, so familiarity is a measured IV rather than an assumed ordering.

## unseen_letter_metric
The seen/unseen split is a genuinely sharp idea and the honest mask (seen_set = letters that ACTUALLY appeared in examples, eval_icl.py) is done right — but the metric has an interpretation-breaking leak and is missing baselines. LEAK (must-fix): with seen_letters=13, examples consume ~13 distinct INPUT letters and therefore ~13 distinct OUTPUT letters; for a bijection the model can narrow an unseen input's image by ELIMINATION (it must map into the ~13 unused outputs) with no rule at all — and elimination improves as coverage grows, contaminating "unseen accuracy = rule use." Worse, elimination helps RANDOM most (it is the only cipher with no other signal), inflating the very floor your argument needs at 0. Fix: drop seen coverage low (6-8 letters, so >=18 unseen and elimination is weak) and REPORT unseen accuracy as a function of seen-fraction; also log per-task actual coverage (how many of 26 appeared) since 6 examples x 6 chars from a 13-pool gives variable, uncontrolled coverage. BASELINES (must-fix, none present): (1) chance line — not naive 1/26=0.038 but the elimination-aware chance (1/#remaining-outputs) so "above chance" means rule use beyond elimination; (2) per-cipher APPLICATION-ONLY run for ALL FOUR kinds (the code supports --application but the arc only mentions it as a control) to establish the EXECUTION ceiling — if random/affine application-only is well below caesar, then low unseen accuracy is partly execution difficulty (long table lookup), not failure to induce; unseen-failure is only evidence for induction-failure if execution is confirmed easy and roughly EQUAL across kinds; (3) rule-stated control (rule in words) for the generalizable ciphers to separate recover-rule-from-examples from execute-rule. COMPREHENSION FLOOR (must-fix, report FIRST): confirm caesar few-shot unseen >> chance; if it floors, the whole battery is uninformative. METRIC ROBUSTNESS: per-letter accuracy is POSITIONAL (zip of out[i] vs answer[i]); one dropped/extra emitted letter misaligns all downstream positions and scores a correct mapping as wrong — report length-match rate and align (or restrict per-letter scoring to correct-length outputs); this penalizes think/random disproportionately. Also report per-unseen-LETTER accuracy (only 13 unseen letters exist, sampled 3/task) to verify the signal isn't driven by a few easy letters, and report variance across cipher KEYS (multiple caesar/affine/random keys with CIs) — atbash currently is n=1 cipher.

## Confounds
- No novel-AND-generalizable cell: affine/atbash/caesar are all named, likely-pretrained ciphers, so the design can never observe induction-of-a-novel-rule; the headline claim is untestable without adding e.g. scrambled-alphabet caesar
- Simplicity/description-length confounds familiarity end-to-end: caesar is shortest-program AND most familiar AND adjacency-preserving (eyeball-inducible), so a familiarity-ordered result is equally explained by easy INDUCTION (opposite conclusion)
- Elimination leak: seen_letters=13 lets the model narrow unseen images by exclusion (bijection) with no rule; inflates unseen accuracy, most for random, contaminating 'unseen = rule only'
- atbash has zero key entropy (single fixed cipher) while caesar/affine/random are distributions — not apples-to-apples; atbash is one point with no variance
- 'Matched complexity' equivocates: matched as output bijections but unmatched on key entropy (~4.5 vs 0 vs 7 vs 88 bits), program length, and inducibility
- Execution vs induction not separated without per-kind application-only: low unseen accuracy on affine/random may be table-lookup execution difficulty, not failure to induce
- Positional per-letter scoring misaligns on length errors, penalizing think/random conditions and understating mappings that are correct but shifted
- Familiarity is assumed, never measured for this specific model — the independent variable (familiarity ordering) is unquantified
- Tokenization: 6-char space-free strings tokenize as subwords, making per-letter reasoning uniformly hard and possibly differentially (English-like vs random substrings); consider space-separated letters to force per-letter tokens
- No chance/CI reporting and single task-seed per condition; caesar k-range (3..23) arbitrarily excludes small shifts and 13, which are the easiest/edge cases

## Must-fix
- Add a NOVEL-but-generalizable cipher that cannot be retrieved but can be induced: caesar-in-a-scrambled-alphabet (fixed random letter ordering P, then shift-by-k within P). Same 1-param description length as caesar; standard-caesar vs scrambled-caesar unseen accuracy is THE crux comparison and isolates familiarity from simplicity. This is required for the design to test its own hypothesis at all.
- Do not treat affine as the novel-generalizable condition — it is a named classical cipher (likely pretrained). Either demote it to a second 'familiar' point or replace with the scrambled-caesar / a keyword cipher with a novel key.
- Run application-only (--application) for ALL FOUR (or more) ciphers and confirm execution is high and roughly EQUAL across kinds; unseen-induction failure is only interpretable if execution is confirmed not to be the bottleneck. Add a rule-stated-in-words control for the generalizable ciphers to separate recover-rule-from-examples from execute-rule.
- Report a comprehension FLOOR first: caesar few-shot unseen accuracy must be well above (elimination-aware) chance, else the whole battery floors and the gradient is meaningless.
- Add and report an elimination-aware chance baseline (1/#unused-outputs, not 1/26), and lower seen coverage (~6-8 letters) so unseen accuracy reflects rule use rather than exclusion; log per-task actual coverage and report unseen accuracy vs seen-fraction.
- Fix key-variance asymmetry: use multiple keys per cipher with CIs across keys and task-seeds; atbash is currently a single fixed cipher (n=1) and cannot be compared to the caesar/affine/random distributions.
- Make per-letter scoring length-robust: report length-match rate and align outputs (or score per-letter only on correct-length outputs) so a single dropped letter doesn't zero out a correct mapping — this differentially penalizes think and random.
- Stop asserting 'matched complexity': either re-frame as matched-bijection-class-but-unmatched-description-length (and lean on the scrambled-caesar pair to control description length), or add the key-entropy/program-length figures explicitly so readers see the gradient is confounded.
- Measure familiarity for THIS model (zero-shot name/apply probes per cipher) so the familiarity ordering is an empirical IV, not an assumption.
- Fill in the empty report.md/README/configs and pre-register the prediction table (per-cell expected unseen accuracy under retrieval vs induction) BEFORE running, since the conclusion hinges on a specific ordering pattern.
